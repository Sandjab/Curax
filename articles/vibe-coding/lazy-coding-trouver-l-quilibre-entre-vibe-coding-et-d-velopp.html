You can still become a senior engineer in vibe coding era
There is this growing fear right now: if junior devs use AI to write code (or just vibe code), they will never become senior (or more experienced) engineer.
We have already seen this notion floating around on X. In a recent conversation with a friend:
I have been tinkering around this from quite some time. In fact more around how junior devs can bring up a taste if they are just getting started.
If you look it more closely, the problem is more about outsourcing judgement than ai-assisted coding itself.
A junior engineer can vibe code (or even code manually) and still stay junior for years if they never learn how to reason about tradeoffs + debug production issues + improve code health over time. There are more parameters to it. A senior or more experienced engineer, on the other hand, can use AI heavily and still be very much senior because the seniority is more about taste, judgment, and responsibility. (A quite good number of L5s/CTOs don't write code often).
I think it is subjective to individual how they want to approach building software and think about systems.
In this article, I am increasingly pushing towards a middle path: let's say it lazy coding.
The first question which arises here is: what differentiates it from vibe coding and standard coding and how it balances the same?
Yes. It is not vibe coding where you only throw prompts (about what your ultimate goal is) and hope it comes out to be true. Also, it is not traditional coding where you THINK too much and might over engineer the first draft. It might slow down the process of building.
WHAT YOU NEED TO DO?
Always have a self belief that you will be learning a lot on the fly.
Try to write a rough version yourself. Even if you don't know, learn to write a rough version. It helps a lot, at least in understanding fundamentals of building software.
Think about systems. Treat it like a black box which you are going to nurture with a lot of iterations. One quite tip here is: "documenting dot md". Ask your model to keep it running while you are doing changes in code. Review it often.
Use models to handle the repetitive parts: refactors, edge cases, tests, cleanup, docs etc.
That rough code is often a better spec than English sometimes. It gives AI context in the format it understands best because practically your prompt can be vague for the model (again, it depends on the task). But you still own the thinking.
And that ownership is what grows you.
The manual effort in building software is changing by an order-of-magnitude and we can quite expect this trend will going to continue only.
But here is a thing!
EVERY STARTUP/BIG TECH/RESEARCH LAB still need a great engineer with great experience in building systems. And they will need them for an horizon of time. I am not sure until when.
Anthropic's career page. LOL!
As builders/researchers/developers, we are supposed to build a system. You can term it as a product or tool or a complex application. Talking with some great engineers + collecting insights + from own experiences - this led me to some habitual thoughts which are quite pragmatic.
  • Explain before you accept. I know this might be tiring as your genie is smart enough to just do things how you want. But let's say, put a count. You will be reviewing that at least 6 out of 10 times. It's reasonable to do it.
  • Try to debug manually at least once. It is actually a quite learning experience given you are building any kind of software.
  • Read the diff. Again tiring, but yeah. This is something your own senior engineer review what is changed (not just what it produced).
  • Think about system design, the data model and the failure modes. I think this is something which you always want to own yourself. Ask the model to help you in that but let a mental model about the system architecture always with you.
I think there is some little things we can do to elevate ourselves till the time AGI is there. This article is a kind of personification of the same. I am still thinking we can bring more steps or methodologies here. Would love to know your thoughts!
