MCP: the only guide you need to connect Claude to everything
I was copying and pasting everything. CSV into Claude, explain what it is, get the output, copy it back into my editor. screenshot of a dashboard, paste it in, ask Claude to analyse it. every conversation started with me manually feeding context that already existed somewhere on my machine.
MCP eliminates that entire loop. it stands for Model Context Protocol, an open standard that lets Claude connect directly to your files, your GitHub repos, your database, your browser. no custom integrations, no API wrappers, no middleware. one protocol, universal access.
Anthropic launched it in November 2024. OpenAI adopted it months later. Google, Microsoft, AWS, and Cloudflare backed it through the Agentic AI Foundation under the Linux Foundation. this is now the industry standard for connecting AI to external systems. and most people still haven't set it up.

What MCP Actually Is

the simplest way to think about it: MCP is a USB-C port for AI. before USB-C, every device had its own charger, its own cable, its own connector. MCP does the same thing for AI integrations, one standardised protocol that works across tools, data sources, and services.
the architecture has three parts:
the host is your AI application, Claude Desktop, Claude Code, or any MCP-compatible client. this is where you interact with the model.
the client sits inside the host and handles communication. it translates your requests into MCP format and routes them to the right server.
the server is the external tool or data source. a filesystem server lets Claude read your local files. a GitHub server lets it manage repos and PRs. a Postgres server lets it query your database.
each server exposes specific capabilities through three primitives:
  • resources: data retrieval. the server returns information but doesn't change anything.
  • tools: actions with side effects. querying an API, writing a file, executing a command.
  • prompts: reusable templates for common workflows.
under the hood, it runs on JSON-RPC 2.0, stdio for local connections, SSE (Server-Sent Events) for remote ones. you don't need to understand the protocol layer to use it. you just need to know how to configure a server.

The Before And After

this is easier to explain with a concrete example.
before MCP: I want Claude to help me write a README for a project. I open Claude, paste in the folder structure, paste in the main config file, paste in a few code snippets for context, explain what the project does. Claude writes a README. I copy it, open my editor, paste it in, save the file. next time I update the project, I do it all again.
after MCP: I tell Claude "write a README for this project." Claude reads the folder structure directly from my filesystem, scans the relevant files, understands the codebase, and writes the README, which I can save directly to the project folder. no pasting, no re-explaining, no manual file juggling.
and this isn't limited to files. with the right MCP servers running, Claude can: read and manage your GitHub repos, PRs, and issues / query your Postgres or Supabase database directly / search the web in real time / automate browser actions (click, screenshot, navigate) / pull messages from Slack...
all through the same standardised protocol. add a server, restart Claude, and the capability is just there.

How To Set It Up

there are two ways to add MCP servers to Claude Desktop: desktop extensions and manual JSON configuration. I'll walk through the manual setup because that's what gives you full control.
STEP 1: find your config file
Claude Desktop stores MCP configuration in a single JSON file.
→ macOS:
bash
~/Library/ApplicationSupport/Claude/claude_desktop_config.json
→ Windows:
bash
%APPDATA%\Claude\claude_desktop_config.json
the fastest way to open it: Settings → Developer → Edit Config inside Claude Desktop.
STEP 2: add a server
each MCP server is an entry in the mcpServers object. here's what a basic filesystem server looks like:
json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/yourname/Projects"
      ]
    }
  }
}
this gives Claude read/write access to your /Projects folder. replace the path with whatever directory you want Claude to see.
prerequisite: most MCP servers require Node.js. if you don't have it, install the LTS version from
nodejs.org
before continuing.
STEP 3: restart Claude Desktop
config changes only take effect after a full restart. on macOS, right-click the dock icon > Quit. on Windows, right-click the system tray icon > Exit. then reopen.
STEP 4: verify it works
look for the hammer icon (🔨) in the Claude Desktop input area. click it, you should see the tools your MCP server exposes. if they're listed, you're connected.
if something's broken, check Settings > Developer > Logs for error messages. 90% of issues are a typo in the config file or a missing Node.js installation.

Set Up The Servers That Actually Matter

you can run multiple servers simultaneously, just add each one as a separate entry in the config. here are the five that cover most use cases, with the exact JSON config for each.
filesystem - Claude reads and writes your local files
json
"filesystem": {
  "command": "npx",
  "args": [
    "-y",
    "@modelcontextprotocol/server-filesystem",
    "/Users/yourname/Projects"
  ]
}
what it unlocks: Claude can read any file in the specified directory, create new files, edit existing ones, and search across your project. this is the foundation, install this first.
tip: don't expose your entire home directory. point it at a specific project folder or workspace. you control the blast radius.
GitHub - repos, PRs, and issues without leaving Claude
json
"github": {
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-github"],
  "env": {
    "GITHUB_PERSONAL_ACCESS_TOKEN": "your_token_here"
  }
}
setup: generate a personal access token at
github.com
> Settings > Developer settings > Personal access tokens. give it repo scope at minimum.
what it unlocks: Claude can create and review PRs, search issues, read repo contents, and manage branches. I use this to draft PR descriptions and scan open issues without context-switching to the browser.
Brave Search - real-time web access for Claude
json
"brave-search": {
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-brave-search"],
  "env": {
    "BRAVE_API_KEY": "your_api_key_here"
  }
}
setup: get a free API key at
brave.com/search/api
. the free tier gives you 2,000 queries/month.
what it unlocks: Claude can search the web in real time and pull current information. essential for research tasks where Claude's training data isn't enough.
Playwright - browser automation
json
"playwright": {
  "command": "npx",
  "args": ["-y", "@executeautomation/playwright-mcp-server"]
}
what it unlocks: Claude can navigate to URLs, take screenshots, click elements, fill forms, and extract page content. useful for scraping, testing, and any workflow that involves interacting with web pages programmatically.
Postgres - Claude queries your database directly
json
"postgres": {
  "command": "npx",
  "args": [
    "-y",
    "@modelcontextprotocol/server-postgres",
    "postgresql://user:password@localhost:5432/mydb"
  ]
}
what it unlocks: Claude can run SQL queries against your database, explore schemas, and pull data for analysis - without you exporting CSVs or screenshots. if you use Supabase, swap the connection string for your Supabase Postgres URL.
warning: use a read-only database user. never give Claude write access to a production database unless you fully understand the implications.
your full config
here's what your claude_desktop_config.json looks like with all five servers:
json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/yourname/Projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_PERSONAL_ACCESS_TOKEN": "your_token_here" }
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": { "BRAVE_API_KEY": "your_api_key_here" }
    },
    "playwright": {
      "command": "npx",
      "args": ["-y", "@executeautomation/playwright-mcp-server"]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://user:password@localhost:5432/mydb"]
    }
  }
}
save the file, restart Claude Desktop, check the hammer icon. you should see tools from all five servers.

The Security Problem We Need To Talk About

MCP is powerful precisely because it gives Claude access to real systems. that same power creates a real attack surface.
  • tool poisoning. MCP servers expose tool descriptions that Claude reads to understand what each tool does. attackers can hide malicious instructions in those descriptions, invisible to you, visible to the model. a server that looks like a legitimate GitHub integration could silently instruct Claude to exfiltrate data from your repos.
  • rug pulls. MCP tools can mutate their definitions after installation. you approve a safe-looking server on day 1. by day 7, the tool descriptions have quietly changed to redirect your API keys to an attacker. there's no built-in alert system for definition changes.
  • supply chain attacks. you're installing npm packages and trusting them with access to your filesystem, your database, your GitHub token. in 2025, a malicious package masquerading as a "Postmark MCP Server" was caught silently BCC'ing all email communications to an attacker. the mcp-remote package (437K+ downloads) had a critical command injection vulnerability (CVE-2025-6514).
  • prompt injection through connected tools. a crafted GitHub issue, just text in a public repo, was shown to hijack the official GitHub MCP server and make Claude pull data from private repos.
how to protect yourself:
vet every server before installing. read the source. check the npm package history.
use read-only credentials wherever possible. especially for databases and GitHub.
don't expose your entire filesystem. scope access to specific directories.
monitor for tool description changes - if a server updates, review what changed before restarting Claude.
treat MCP servers like browser extensions: every one you add expands your attack surface.

Limitations - What To Know Before Uou Go All In

local only (mostly). MCP servers run on your machine. close Claude Desktop, they stop. there's no persistent background service unless you set one up yourself.
context overhead. every connected server adds tool descriptions to Claude's context. run too many servers simultaneously and you eat into your available context window. 2026 updates added lazy loading to help with this, but it's still worth being selective.
no universal auth standard. each server handles authentication differently. some use API keys, some use OAuth, some use connection strings. there's no unified credential management across servers.
you're trusting npm. most servers are npm packages maintained by individual developers or small teams. there's no Anthropic-curated "app store" with security reviews. desktop extensions are starting to fill this gap, but the ecosystem is still largely trust-based.

The Bottom Line

MCP is the difference between an AI that answers questions and an AI that's wired into how you actually work. it turns Claude from a text box into a tool that reads your files, queries your databases, manages your repos, and searches the web, all through one standardised protocol.
start with the filesystem server. see how it feels when Claude can actually read your project instead of you pasting files in. then add GitHub, then Brave Search, then whatever makes sense for your workflow. five servers cover 80% of use cases.
but don't skip the security section. every server you add is a trust decision. vet what you install, scope your permissions, use read-only credentials. the productivity gain is real. so is the attack surface.
