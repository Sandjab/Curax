How AI Impacts Skill Formation

Vulgarisation

Quand l'IA code à votre place, qu'apprenez-vous vraiment ?

Un développeur junior rejoint une équipe. On lui confie sa première tâche : intégrer une bibliothèque Python qu'il n'a jamais utilisée. Il ouvre son assistant IA, colle les instructions, récupère le code généré, l'exécute. Ça marche. Tâche terminée en vingt minutes. Mais deux semaines plus tard, quand un bug survient dans ce même code, il est incapable de le diagnostiquer. Ce scénario, Judy Hanwen Shen et Alex Tamkin, chercheurs chez Anthropic, l'ont reproduit en laboratoire pour en mesurer les effets réels.

Leur étude, publiée en février 2026, pose une question que beaucoup de managers et d'ingénieurs préfèrent esquiver : les gains de productivité offerts par l'IA se paient-ils par une érosion des compétences techniques ? La réponse, étayée par une expérience contrôlée sur 52 développeurs professionnels, est sans ambiguïté.

Le défi : mesurer ce que l'IA fait à nos compétences

Depuis 2023, les études sur la productivité assistée par IA se multiplient. GitHub Copilot accélère l'écriture de code de 26 à 55 % selon les contextes. Les consultants équipés de GPT-4 terminent 12 % de tâches en plus. Les agents de centres d'appel résolvent 15 % de tickets supplémentaires. Le constat semble unanime : l'IA rend plus productif, et les travailleurs les moins expérimentés en profitent le plus.

Mais ces études mesurent toutes la même chose : le produit fini. Combien de lignes de code, combien de tickets fermés, combien de tâches cochées. Personne ne regardait ce qui se passe dans la tête du développeur pendant qu'il utilise l'IA. Acquiert-il les compétences nécessaires pour, demain, superviser le code que l'IA écrit à sa place ? Ou bien délègue-t-il sa propre formation ?

La question n'est pas académique. Dans les systèmes critiques — finance, santé, transport — un humain reste responsable de valider le code avant déploiement. Si cet humain n'a jamais appris à lire et débugger le code qu'il est censé superviser, la chaîne de sécurité se brise.

L'expérience : apprendre une bibliothèque Python, avec ou sans IA

Shen et Tamkin ont conçu un protocole rigoureux, affiné au fil de quatre études pilotes. Le principe : demander à des développeurs professionnels d'apprendre la bibliothèque Python Trio (programmation asynchrone), puis évaluer ce qu'ils ont retenu.

Trio a été choisie pour des raisons précises. Elle est peu connue — bien moins que son équivalent asyncio — ce qui garantit que les participants partent tous du même niveau. Elle introduit des concepts spécifiques (nurseries, concurrence structurée) qui ne se devinent pas à partir de connaissances Python générales. Et elle est conçue pour être accessible, ce qui en fait un bon sujet d'apprentissage en temps limité.

52 développeurs, tous avec au moins un an d'expérience Python et une pratique hebdomadaire du langage, ont été répartis aléatoirement en deux groupes. Le groupe traitement disposait d'un assistant IA (basé sur GPT-4o) capable de générer du code complet, répondre aux questions conceptuelles et débugger. Le groupe contrôle travaillait seul, avec les instructions et la documentation.

Les deux groupes avaient 35 minutes pour compléter deux tâches de programmation asynchrone. Ensuite, tous passaient le même quiz de 14 questions couvrant trois compétences : compréhension conceptuelle, lecture de code et debugging. L'IA était interdite pendant le quiz.

Les chercheurs ont aussi enregistré l'écran de chaque participant — une précaution née des pilotes, où 25 à 35 % des participants du groupe contrôle trichaient en utilisant l'IA en cachette. Quatre itérations du protocole ont été nécessaires pour éliminer ce problème.

Un mot sur la rigueur méthodologique

L'étude a été pré-enregistrée (protocole et grille de notation déposés avant l'expérience). Les participants, recrutés via une plateforme de crowd-working, étaient rémunérés 150 dollars pour environ une heure de travail. Les deux groupes étaient équilibrés sur l'expérience en programmation, la fréquence d'utilisation de Python et la familiarité avec la programmation asynchrone. Un échauffement préalable (tâche Python sans rapport avec Trio) permettait de calibrer le niveau de base.

Les résultats : productivité stable, compétences en chute

Premier résultat, contre-intuitif : l'IA n'a pas rendu les développeurs significativement plus rapides. Le temps moyen de complétion était similaire dans les deux groupes — autour de 23-24 minutes. Ce résultat contredit les études antérieures qui rapportaient des accélérations de 26 à 55 %.

L'explication se trouve dans les enregistrements d'écran. Certains participants passaient jusqu'à 11 minutes à rédiger leurs requêtes à l'assistant IA — relisant les instructions, reformulant leurs questions, attendant les réponses. Sur une tâche de 35 minutes, ce temps d'interaction avec l'IA compensait presque exactement le temps gagné sur l'écriture du code. Seuls ceux qui collaient directement le code généré par l'IA sans réfléchir terminaient nettement plus vite (19,5 minutes contre 23 minutes).

Deuxième résultat, le plus frappant : les scores au quiz divergent fortement. Le groupe avec IA obtient en moyenne 17 points de pourcentage de moins que le groupe sans IA. En termes scolaires, c'est l'écart entre un B et un D. L'effet est statistiquement robuste (Cohen's d = 0,738, p = 0,01) et persiste après ajustement pour le niveau de base des participants.

La décomposition par type de question révèle où le bât blesse. L'écart le plus large concerne le debugging : les participants assistés par IA sont nettement moins capables d'identifier et de diagnostiquer des erreurs dans le code. Les questions de lecture de code montrent un écart plus faible — logique, puisque les deux groupes ont été exposés à du code pendant la tâche. Les questions conceptuelles se situent entre les deux.

Ce résultat s'explique par une asymétrie dans l'exposition aux erreurs. Le participant médian du groupe contrôle a rencontré trois erreurs pendant la tâche ; celui du groupe IA, une seule. Dans le groupe contrôle, ces erreurs étaient souvent liées aux concepts fondamentaux de Trio — coroutines non attendues, fonctions asynchrones mal typées — et leur résolution forçait une compréhension en profondeur. Le groupe IA, lui, voyait du code fonctionnel du premier coup.

Six façons d'utiliser l'IA, trois qui préservent l'apprentissage

L'analyse qualitative des enregistrements d'écran a permis d'identifier six patterns d'interaction avec l'assistant IA, avec des résultats radicalement différents sur l'apprentissage.

Trois patterns produisent des scores faibles (24 à 39 % au quiz) :

  • La délégation totale : le participant demande à l'IA de générer tout le code, le colle tel quel. C'est le pattern le plus rapide (19,5 minutes en moyenne) mais le plus destructeur pour l'apprentissage (39 % au quiz).
  • La dépendance progressive : le participant commence par poser quelques questions, puis bascule vers la génération de code intégrale pour la deuxième tâche. Il ne maîtrise aucun concept de la seconde partie.
  • Le debugging itératif par IA : le participant demande à l'IA de vérifier et corriger son code à répétition, sans chercher à comprendre les erreurs. Ce pattern cumule lenteur (31 minutes) et faible apprentissage (24 %).

Trois patterns préservent l'apprentissage (65 à 86 % au quiz) :

  • L'interrogation conceptuelle : le participant ne demande jamais de code. Il pose des questions sur le fonctionnement de Trio, puis écrit lui-même. Ce groupe, le plus large (7 participants), obtient 68 % au quiz avec un temps de complétion de 24 minutes — aussi rapide que la délégation totale, mais avec un apprentissage réel.
  • La génération puis compréhension : le participant demande d'abord du code, puis pose des questions pour comprendre ce que l'IA a produit. Score le plus élevé : 86 %.
  • L'hybride code-explication : chaque requête combine une demande de code et une demande d'explication. Le temps de lecture des explications allonge la tâche, mais le score atteint 65 %.

Le dénominateur commun des patterns efficaces : l'engagement cognitif. Le participant réfléchit, questionne, confronte sa compréhension au code — qu'il l'ait écrit lui-même ou non.

Ce que ça change pour l'industrie

Ces résultats arrivent à un moment où les entreprises tech accélèrent l'adoption d'assistants IA pour leurs équipes de développement. La promesse est séduisante : des développeurs juniors immédiatement productifs grâce à l'IA. L'étude de Shen et Tamkin suggère que cette productivité immédiate peut se transformer en dette de compétences.

Le parallèle avec d'autres domaines est éclairant. Des chercheurs ont montré que les professionnels de santé formés avec l'assistance de l'IA développent moins bien leurs capacités de diagnostic visuel. Des consultants en data science voient leurs compétences techniques disparaître dès que l'accès à l'IA est retiré — un effet qualifié d'« exosquelette » par les auteurs d'une étude de 2024. Les compétences augmentées par l'IA ne se transfèrent pas à l'humain.

Pour les organisations, la conclusion pratique est double. D'abord, le debugging est la compétence la plus menacée — précisément celle dont on a le plus besoin pour superviser du code généré par IA. Un ingénieur qui n'a jamais debuggé manuellement ne saura pas repérer les erreurs subtiles d'un agent de code autonome. Ensuite, la manière dont l'IA est utilisée compte autant que le fait de l'utiliser. Interdire l'IA pendant la formation n'est pas la seule option : encourager les patterns à fort engagement cognitif (questions conceptuelles, demandes d'explication) peut préserver l'apprentissage tout en bénéficiant de l'assistance.

Les participants eux-mêmes en étaient conscients. Dans les retours libres après l'expérience, ceux du groupe IA écrivaient qu'ils se sentaient « paresseux », qu'ils auraient dû « lire le code généré plus attentivement », qu'il restait « beaucoup de lacunes dans leur compréhension ». Ceux du groupe contrôle décrivaient la tâche comme « fun » et estimaient que les instructions les avaient aidés à comprendre Trio. Le ressenti subjectif confirmait les scores objectifs.

Un détail supplémentaire mérite attention : copier manuellement le code généré par l'IA (plutôt que de le coller directement) ne produisait aucune amélioration de l'apprentissage. Le temps passé à taper ne remplace pas le temps passé à réfléchir. L'effort cognitif prime sur l'effort mécanique.

Questions ouvertes et pistes futures

L'étude porte sur une seule bibliothèque, apprise en moins d'une heure. La formation professionnelle réelle s'étale sur des mois. L'effet mesuré ici pourrait s'amplifier avec le temps (accumulation de lacunes) ou au contraire s'atténuer (les développeurs finissent par apprendre malgré tout à force de répétition). Seules des études longitudinales en entreprise permettront de trancher.

L'interface utilisée était un chat textuel, où le développeur devait formuler ses requêtes. Les outils de coding agentiques actuels — qui modifient le code directement dans l'éditeur, sans que l'utilisateur ne formule de question — réduisent encore la friction cognitive. Les auteurs estiment que l'effet négatif sur l'apprentissage serait probablement amplifié avec ces outils, puisque même le temps de réflexion lié à la rédaction des requêtes disparaît.

L'étude ne compare pas non plus l'assistance IA à l'assistance humaine. Un mentor ou un collègue en pair programming guide l'apprentissage différemment d'un chatbot — avec des questions socratiques, des refus de donner la réponse, une adaptation au niveau perçu. Comparer ces deux formes d'assistance serait une extension naturelle.

Enfin, les modes d'apprentissage intégrés aux assistants IA (comme le mode « Study » de ChatGPT ou le mode « Learning » de Claude Code) représentent une piste concrète. Ces modes pourraient refuser de générer du code complet et privilégier les explications, reproduisant artificiellement les patterns à fort engagement cognitif identifiés dans l'étude. Reste à vérifier empiriquement si ces garde-fous fonctionnent en pratique, ou si les utilisateurs pressés par le temps les désactivent systématiquement.

Le message de fond de cette recherche tient en une phrase : la productivité assistée par IA n'est pas un raccourci vers la compétence. Les organisations qui misent tout sur la vitesse d'exécution sans protéger les parcours d'apprentissage de leurs équipes construisent une dépendance technique dont le coût se révélera quand l'IA se trompera — et qu'il n'y aura personne pour s'en apercevoir.
