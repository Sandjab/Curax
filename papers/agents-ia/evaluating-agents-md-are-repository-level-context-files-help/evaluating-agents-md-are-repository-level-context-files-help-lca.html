Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?

Lecture Critique d'Article

Objectif et contexte

Les fichiers de contexte — AGENTS.md, CLAUDE.md et équivalents — se sont imposés comme pratique standard pour adapter les agents de codage aux spécificités d'un dépôt logiciel. Plus de 60 000 dépôts GitHub publics en contenaient au moment de la rédaction. Les principaux fournisseurs d'agents (Anthropic, OpenAI, Alibaba) recommandent leur utilisation et proposent des commandes dédiées pour les générer automatiquement. Aucune étude rigoureuse n'avait cependant mesuré si ces fichiers améliorent réellement la capacité des agents à résoudre des tâches d'ingénierie logicielle.

Cette publication d'ETH Zurich et LogicStar.ai comble ce manque. Les auteurs évaluent l'effet des fichiers de contexte sur quatre agents de codage (Claude Code, Codex, Qwen Code) dans trois configurations : absence de fichier, fichier généré par LLM, fichier écrit par un développeur humain. L'évaluation couvre SWE-BENCH LITE (300 tâches, 11 dépôts populaires) et un nouveau benchmark, AGENTBENCH (138 instances, 12 dépôts de niche contenant des fichiers de contexte écrits par des développeurs).

Méthodologie

Construction d'AGENTBENCH

Le benchmark AGENTBENCH suit un pipeline en cinq étapes. La sélection cible des dépôts Python contenant un fichier de contexte à la racine, disposant d'une suite de tests et comptant au moins 400 pull requests. Les PR sont filtrées par un mélange de règles heuristiques et d'un agent LLM qui évalue leur caractère déterministe et testable. Un agent configure l'environnement d'exécution, un autre génère des descriptions de tâches standardisées à partir des PR et issues associées, et un dernier produit les tests unitaires de régression. Les tests générés sont vérifiés manuellement pour éliminer la sur-spécification.

Le choix d'utiliser GPT-5.2 avec Codex comme agent de construction introduit une dépendance à un modèle spécifique pour la qualité des instances. La couverture moyenne des tests atteint 75 % du code modifié — un chiffre correct mais qui laisse une marge d'erreur non négligeable sur le quart restant.

Protocole expérimental

Quatre agents sont testés, chacun couplé à son modèle natif : Claude Code avec Sonnet 4.5, Codex avec GPT-5.2 et GPT-5.1 Mini, Qwen Code avec Qwen3-30B-Coder. Les températures sont fixées à 0 pour trois agents sur quatre (0.7 pour Qwen, avec top-p 0.8). Chaque configuration est exécutée une seule fois par instance — une limitation significative en l'absence d'estimation de variance.

Les trois conditions expérimentales (NONE, LLM, HUMAN) sont clairement définies. Pour la condition LLM, chaque agent génère son propre fichier via sa commande d'initialisation recommandée. Les fichiers développeur proviennent de l'état pré-patch du dépôt. Les agents ont accès au web mais une vérification manuelle confirme l'absence de triche (consultation des PR correspondantes).

Résultats principaux

Le résultat central contredit les recommandations des fournisseurs d'agents. Les fichiers de contexte générés par LLM diminuent le taux de résolution de 0,5 % sur SWE-BENCH LITE et de 2 % sur AGENTBENCH en moyenne. Les fichiers écrits par des développeurs améliorent la performance de 4 % en moyenne sur AGENTBENCH, mais cette amélioration reste marginale et ne se manifeste pas pour tous les agents — Claude Code fait exception avec une dégradation.

Les coûts augmentent systématiquement. Les fichiers LLM ajoutent en moyenne 2,45 étapes sur SWE-BENCH LITE et 3,92 sur AGENTBENCH, soit une hausse de coût de 20 à 23 %. Le nombre de tokens de raisonnement augmente de 10 à 22 % selon les modèles. Les agents consacrent plus de temps à explorer le dépôt, lancer des tests et utiliser des outils spécifiques mentionnés dans le fichier de contexte — sans que cet effort supplémentaire se traduise en résultats.

Un résultat éclairant : lorsque toute la documentation existante est retirée du dépôt, les fichiers de contexte LLM surpassent les fichiers humains de 2,7 % en moyenne. Les fichiers générés sont donc redondants avec la documentation existante, mais utiles quand cette documentation manque — ce qui correspond aux témoignages anecdotiques de développeurs de projets peu documentés.

Ni le modèle utilisé pour la génération, ni le prompt employé n'ont d'effet significatif et cohérent sur la qualité des fichiers produits. GPT-5.2 génère des fichiers qui améliorent les performances sur SWE-BENCH LITE (+2 %) mais les dégradent sur AGENTBENCH (-3 %).

Discussion et limites

La restriction à Python constitue la limitation la plus sérieuse. Ce langage, surreprésenté dans les données d'entraînement, implique que les modèles possèdent déjà une connaissance paramétrique de l'outillage et des dépendances des dépôts populaires. L'effet des fichiers de contexte pourrait différer pour des langages moins courants (Rust, Haskell, OCaml) où cette connaissance préalable est plus faible.

L'exécution unique par configuration empêche toute analyse statistique de la variance. Avec des taux de résolution oscillant entre 30 et 60 %, une différence de 2 à 4 points de pourcentage pourrait relever du bruit, surtout sur 138 instances. L'absence de tests de significativité ou d'intervalles de confiance affaiblit les conclusions quantitatives.

AGENTBENCH ne contient que 12 dépôts. La distribution, bien que plus homogène que SWE-BENCH LITE, reste trop restreinte pour garantir la généralisation. Certains dépôts ne contribuent que 3 instances (graphiti), rendant les estimations par dépôt instables.

L'évaluation se limite au taux de résolution. Les fichiers de contexte pourraient améliorer d'autres dimensions — qualité du code produit, conformité aux conventions du projet, sécurité — que cette étude ne mesure pas. Les auteurs reconnaissent cette limitation et suggèrent des pistes pour de futurs travaux.

Le protocole de génération des tests de régression par LLM, malgré la vérification manuelle, laisse subsister un risque de tests insuffisamment discriminants. La couverture moyenne de 75 % signifie qu'un quart du code modifié n'est pas vérifié.

Reproductibilité

Le code de génération d'AGENTBENCH et d'évaluation est annoncé comme disponible publiquement. Tous les prompts utilisés pour le filtrage des PR, la configuration des environnements, la génération des descriptions de tâches et des tests sont reproduits intégralement en annexe. Les versions des modèles sont spécifiées (Sonnet 4.5, GPT-5.2, GPT-5.1 Mini, Qwen3-30B-Coder).

Reproduire les expériences nécessite un budget d'inférence conséquent : Claude Code avec Sonnet 4.5 coûte en moyenne 1,30 USD par instance (40 instances par dépôt SWE-BENCH LITE), soit plusieurs milliers de dollars pour l'ensemble des configurations. Le déploiement local de Qwen3-30B via vLLM requiert du matériel GPU dédié. Ces contraintes pratiques limitent la reproductibilité à des équipes disposant de ressources significatives.

Impact et applications

Cette étude remet en question une pratique largement adoptée par l'industrie. La recommandation pratique est claire : ne pas utiliser de fichiers de contexte générés automatiquement par LLM dans l'état actuel, et limiter les fichiers manuels à des instructions minimales (outillage spécifique, commandes de build). Les survols de codebase, que 95 à 100 % des fichiers générés incluent, n'aident pas les agents à localiser plus rapidement les fichiers pertinents.

Le benchmark AGENTBENCH enrichit l'écosystème d'évaluation avec des dépôts de niche, complémentaires aux grands projets de SWE-BENCH LITE. Cette diversification est bienvenue, les benchmarks existants étant dominés par quelques dépôts populaires (django représente la majorité des instances SWE-BENCH LITE).

L'analyse des traces révèle un phénomène contre-intuitif : les agents suivent les instructions des fichiers de contexte (utilisation d'outils mentionnés, exécution de tests recommandés), mais ce suivi accroît la complexité de la tâche sans améliorer le résultat. Suivre plus d'instructions rend la résolution plus difficile, pas plus facile.

Positionnement dans la littérature

Cette publication se distingue des travaux descriptifs antérieurs (Chatlatanagulchai et al., 2025 ; Mohsenimofidi et al., 2025) qui cataloguaient le contenu des fichiers de contexte sans évaluer leur efficacité. Elle s'inscrit dans la lignée des benchmarks d'évaluation des agents de codage initiée par SWE-BENCH (Jimenez et al., 2024) et étendue vers la génération de fonctionnalités (FEA-Bench), la sécurité (SecureAgentBench) et la performance (SWE-Perf).

Le résultat principal — l'inefficacité des fichiers de contexte générés automatiquement — entre en tension directe avec les recommandations de GitHub (Nigh, 2025), Anthropic (2025b) et les témoignages de développeurs individuels (Sewell, 2025). L'explication proposée (redondance avec la documentation existante) est convaincante mais partielle : elle n'explique pas pourquoi les fichiers augmentent la difficulté perçue par les modèles, mesurée par le nombre de tokens de raisonnement.

La direction future la plus prometteuse concerne la génération de fichiers de contexte ciblés et minimaux, s'appuyant sur les travaux en planification (Dynamic Cheatsheet, Suzgun et al., 2025) et en ingénierie de contexte agentique (Zhang et al., 2025b). L'écart documenté entre recommandations industrielles et résultats empiriques crée un espace de recherche concret pour améliorer l'auto-adaptation des agents à leurs environnements de travail.
