Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?

Lecture Critique d'Article

1. Objectif et contexte

Les fichiers contexte — AGENTS.md, CLAUDE.md — se sont imposés comme pratique standard pour adapter les agents de code aux dépôts logiciels. Plus de 60 000 dépôts GitHub en intègrent un au moment de la rédaction de l'article. OpenAI, Anthropic et Alibaba recommandent cette approche et fournissent des commandes dédiées (/init) pour générer automatiquement ces fichiers. Malgré cette adoption massive, aucune évaluation empirique n'avait mesuré leur effet réel sur la capacité des agents à résoudre des tâches d'ingénierie logicielle.

Gloaguen et al. posent une question frontale : ces fichiers aident-ils réellement ? Pour y répondre, ils construisent un nouveau benchmark (AGENTBENCH, 138 instances sur 12 dépôts Python) et évaluent quatre agents de code sous trois conditions — sans fichier contexte, avec fichier généré par LLM, avec fichier écrit par les développeurs du dépôt.

2. Méthodologie

Construction d'AGENTBENCH

Le benchmark repose sur un pipeline en cinq étapes. La sélection cible des dépôts Python contenant un fichier contexte à la racine, disposant d'une suite de tests et comptant au moins 400 pull requests. Douze dépôts satisfont ces critères, parmi lesquels smolagents, openai-agents-python, fastmcp, tinygrad et ragas.

Les PRs sont filtrées par combinaison de règles (référence à une issue, modification d'au moins un fichier Python) et d'un agent LLM qui évalue si la PR introduit un comportement déterministe et testable. Contrairement à SWE-BENCH LITE, les PRs ne sont pas tenues de contenir des tests unitaires — un choix pragmatique dicté par les pratiques moins strictes des dépôts de niche.

Les descriptions de tâches et les tests unitaires sont générés par GPT-5.2 via Codex. Les descriptions suivent un template standardisé en six sections. Les tests doivent échouer sur l'état pré-patch et réussir après application du patch original. Une vérification manuelle de 10 % des instances n'a révélé aucune fuite de solution. La couverture moyenne des tests générés atteint 75 % du code modifié.

Protocole expérimental

Quatre agents sont évalués, chacun couplé à son modèle natif : Claude Code avec Sonnet-4.5, Codex avec GPT-5.2 et GPT-5.1 Mini, Qwen Code avec Qwen3-30B-Coder. La température est fixée à 0 pour tous sauf Qwen (0,7 avec top-p 0,8). Un seul échantillon est prélevé par instance.

Trois conditions expérimentales sont comparées. NONE : aucun fichier contexte (pour AGENTBENCH, le fichier développeur est supprimé). LLM : fichier généré par la commande d'initialisation recommandée de chaque agent. HUMAN : fichier contexte originel du dépôt (disponible uniquement pour AGENTBENCH).

Les agents opèrent dans des conteneurs Docker avec accès Internet. L'historique git et les remotes sont supprimés pour limiter le risque de triche. Une inspection manuelle confirme qu'aucun agent n'a consulté la PR correspondant à l'instance évaluée.

3. Résultats principaux

Le résultat central est contre-intuitif. Les fichiers contexte générés par LLM réduisent le taux de succès dans 5 des 8 configurations testées — une baisse moyenne de 0,5 % sur SWE-BENCH LITE et 2 % sur AGENTBENCH. Les fichiers écrits par les développeurs améliorent marginalement la performance (4 % en moyenne sur AGENTBENCH), sauf pour Claude Code où elle diminue.

Le coût augmente systématiquement. Les fichiers LLM ajoutent en moyenne 2,45 étapes par instance sur SWE-BENCH LITE et 3,92 sur AGENTBENCH, soit une hausse de coût d'inférence de 20 à 23 %. Les fichiers développeurs produisent un effet comparable : +3,34 étapes, jusqu'à +19 % de coût.

L'analyse des traces révèle que les agents suivent effectivement les instructions des fichiers contexte. L'outil uv est utilisé 1,6 fois par instance quand il est mentionné dans le fichier, contre moins de 0,01 fois sinon. Les outils spécifiques au dépôt passent de 0,05 à 2,5 utilisations. Le problème n'est donc pas un défaut de suivi d'instructions — les agents respectent les consignes, mais ces consignes ne les aident pas à résoudre la tâche.

Les fichiers contexte provoquent une exploration plus large : davantage de tests exécutés, de fichiers lus, de recherches par grep. GPT-5.2 consomme 22 % de tokens de raisonnement supplémentaires avec un fichier LLM sur SWE-BENCH LITE. Les agents traitent les instructions additionnelles comme des contraintes à satisfaire, ce qui complexifie l'exécution sans améliorer le résultat.

Les overviews de codebase inclus dans les fichiers contexte ne réduisent pas le nombre d'étapes avant la première interaction avec un fichier pertinent. 100 % des fichiers générés par Sonnet-4.5 et 99 % de ceux de GPT-5.2 contiennent un overview, mais cette information ne se traduit pas en navigation plus efficace.

Un résultat complémentaire éclaire la nature des fichiers LLM-generated : quand toute la documentation existante est supprimée (fichiers .md, dossier docs/, exemples), les fichiers LLM deviennent bénéfiques (+2,7 % en moyenne) et surpassent les fichiers développeurs. Leur contenu est donc largement redondant avec la documentation déjà présente dans le dépôt.

Ablations

Utiliser un modèle plus puissant (GPT-5.2) pour générer les fichiers contexte améliore la performance sur SWE-BENCH LITE (+2 %) mais la dégrade sur AGENTBENCH (−3 %). Le prompt utilisé pour la génération (Codex vs Claude Code) n'a pas d'effet consistant. Ces deux résultats suggèrent que le problème est structurel — pas une question de qualité de génération.

4. Discussion et limites

La restriction à Python représente le biais le plus lourd. Python est massivement représenté dans les données d'entraînement des LLM — les modèles disposent déjà d'une connaissance paramétrique étendue sur son outillage et ses conventions. L'effet des fichiers contexte pourrait être radicalement différent pour Rust, Haskell ou Zig, où les connaissances intégrées au modèle sont plus lacunaires. Les auteurs reconnaissent cette limitation mais n'offrent aucune donnée préliminaire sur d'autres langages.

L'évaluation repose sur un seul échantillon par configuration. Aucun intervalle de confiance n'est rapporté, aucun test statistique de significativité n'est conduit. Les différences observées — 0,5 à 4 % — sont faibles et pourraient relever de la variance d'échantillonnage, surtout avec un seul run par instance. Cette absence d'analyse statistique affaiblit les conclusions.

AGENTBENCH ne comprend que 12 dépôts. Certains ne contribuent que 3 instances (graphiti) ou 6 (transformers). La granularité par dépôt (Figure 12) montre qu'aucun dépôt individuel ne présente d'effet clair des fichiers contexte — les tendances agrégées masquent une forte hétérogénéité sous-jacente.

Les descriptions de tâches et les tests unitaires sont générés par GPT-5.2 via Codex. Ce recours introduit un biais circulaire : l'évaluation dépend d'artefacts produits par un modèle de la même famille que ceux évalués. La vérification manuelle de non-fuite de solution ne porte que sur 10 % des instances — un taux insuffisant pour exclure toute contamination.

La couverture des tests générés varie de 2,5 % à 100 % (moyenne 75 %). Les instances à faible couverture pourraient accepter des patches incorrects ou rejeter des solutions valides, faussant le taux de succès dans les deux sens.

L'étude ne mesure que la résolution de tâches. Les fichiers contexte pourraient avoir un effet positif sur d'autres dimensions — qualité du code produit, sécurité, respect des conventions de style, maintenabilité — qui échappent à cette évaluation binaire pass/fail.

5. Reproductibilité

Le code source et le benchmark sont publiés. Les prompts utilisés pour chaque étape du pipeline (filtrage de PRs, configuration d'environnement, description de tâches, génération de tests, analyse de traces) sont reproduits intégralement en annexe — un effort de transparence rare dans ce domaine. Les versions de modèles et les paramètres d'inférence sont documentés avec précision. Les environnements Docker sont spécifiés.

La reproductibilité reste cependant partielle. Les modèles propriétaires (GPT-5.2, GPT-5.1 Mini, Sonnet-4.5) ne sont pas accessibles indépendamment : leurs poids ne sont pas publiés et leurs API peuvent évoluer entre deux expériences. Seul Qwen3-30B-Coder est déployable localement via vLLM. Les résultats exacts ne sont donc reproductibles que tant que les API des fournisseurs restent stables — une contrainte classique mais réelle de la recherche sur les LLM propriétaires.

6. Impact et applications

La recommandation pratique est directe : ne pas utiliser de fichiers contexte générés automatiquement par LLM. Pour les fichiers écrits manuellement, se limiter aux exigences minimales — outillage spécifique, commandes de build, contraintes non découvrables par l'agent seul. Toute instruction superflue risque de complexifier la tâche sans bénéfice mesurable.

AGENTBENCH comble un manque dans l'écosystème de benchmarks. Aucun jeu de données existant ne ciblait des dépôts contenant des fichiers contexte écrits par les développeurs. Sa construction à partir de dépôts de niche récents le rend complémentaire de SWE-BENCH LITE, dominé par des projets populaires comme Django et scikit-learn.

L'observation que les fichiers LLM-generated sont redondants avec la documentation existante soulève une question d'ingénierie : les agents bénéficieraient-ils davantage d'une documentation structurée classique (README, docstrings, exemples) que d'un fichier dédié généré par un LLM qui ne fait que reformuler ce qui existe déjà ?

Le décalage entre les recommandations des fournisseurs d'agents et les résultats mesurés par cette étude a des implications commerciales directes. Les commandes /init intégrées dans Claude Code, Codex et Qwen Code génèrent des fichiers qui, selon ces données, dégradent la performance et augmentent les coûts.

7. Positionnement dans la littérature

Deux travaux antérieurs (Chatlatanagulchai et al., 2025 ; Mohsenimofidi et al., 2025) avaient catalogué et catégorisé le contenu des fichiers contexte sans mesurer leur efficacité. Des témoignages individuels (Sewell, 2025 ; Sawers, 2025) rapportaient des améliorations anecdotiques. L'article de Gloaguen et al. est le premier à fournir une évaluation empirique contrôlée.

L'article s'inscrit dans la lignée des benchmarks repository-level initiée par SWE-bench (Jimenez et al., 2024) et étendue à l'ajout de fonctionnalités (FEA-Bench, SWE-Dev), à la génération de tests (SWT-Bench), à la performance du code (SWE-Perf) et à la sécurité (SecureAgentBench). AGENTBENCH se distingue par son focus sur des dépôts de niche récents avec fichiers contexte actifs — un créneau qu'aucun benchmark existant ne couvrait.

Les résultats entrent en tension frontale avec le discours industriel dominant. Anthropic, OpenAI et Qwen recommandent explicitement l'usage de fichiers contexte et intègrent des commandes de génération automatique dans leurs agents. Cette étude montre que ces recommandations ne sont pas soutenues par les données empiriques, du moins pour la résolution de tâches Python sur des dépôts disposant déjà d'une documentation. Le décalage entre recommandations industrielles et résultats mesurés constitue la contribution la plus percutante de l'article.

La piste la plus fertile pour la suite concerne l'amélioration de la génération automatique de fichiers contexte — via l'apprentissage continu à partir de tâches résolues (Zhang et al., 2025b ; Cheng et al., 2025) plutôt que par une analyse statique du dépôt à un instant donné.
