Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?

Vulgarisation

Introduction

Plus de 60 000 dépôts GitHub publics contiennent aujourd'hui un fichier AGENTS.md ou CLAUDE.md — un README destiné non pas aux développeurs humains, mais aux agents de code IA. L'idée paraît logique : si un nouveau développeur a besoin d'un guide pour s'orienter dans un projet, pourquoi pas un agent autonome ? Les éditeurs d'outils — Anthropic, OpenAI, Alibaba — recommandent chaudement cette pratique. Certains intègrent même une commande /init pour générer automatiquement ce fichier contextuel.

Mais personne n'avait vérifié si ces fichiers servaient réellement à quelque chose. Une équipe de l'ETH Zurich et de LogicStar.ai a décidé de poser la question frontalement : les fichiers contextuels améliorent-ils la capacité des agents à résoudre de vrais problèmes de code ? Leur réponse, publiée en février 2026, est contre-intuitive. Les fichiers générés par IA dégradent les performances dans la majorité des cas, et les fichiers rédigés par des humains n'apportent qu'un gain marginal — le tout pour un surcoût de 20 % en inférence.

Le problème

Un agent de code comme Claude Code, Codex ou Qwen Code fonctionne en boucle : il lit un ticket, explore le dépôt, modifie des fichiers, lance des tests, corrige ses erreurs, et recommence jusqu'à produire un patch qui passe la suite de tests. Pour l'aider, on peut lui fournir un fichier contextuel décrivant l'architecture du projet, les conventions de style, les outils de build, la structure des répertoires.

Le problème ? Aucune étude rigoureuse n'avait mesuré l'effet de ces fichiers sur la résolution réelle de tickets. Les témoignages existants se limitaient à des retours anecdotiques de développeurs individuels, du type « mon agent fonctionne mieux depuis que j'ai ajouté un AGENTS.md ». Deux obstacles rendaient l'évaluation difficile. D'abord, les benchmarks classiques comme SWE-bench portent sur des projets populaires (Django, Matplotlib, scikit-learn) qui ne contiennent pas de fichiers contextuels — ces fichiers n'existant que depuis mi-2025. Ensuite, les dépôts qui en possèdent sont des projets récents et de niche, avec des pratiques de documentation hétérogènes et des pull requests souvent peu documentées.

Les chercheurs devaient donc construire un nouveau benchmark, adapté à ces projets de niche, tout en comparant les résultats sur le benchmark de référence existant.

La méthode

L'équipe a travaillé sur deux fronts simultanés. Le premier : évaluer des fichiers contextuels générés par IA sur SWE-bench Lite, le benchmark classique de 300 tickets issus de 11 projets Python majeurs. Le second : créer de toutes pièces un nouveau benchmark, baptisé AgentBench, à partir de projets qui utilisent déjà des fichiers contextuels rédigés par leurs développeurs.

Construire AgentBench

La construction d'AgentBench ressemble à un entonnoir. Les chercheurs ont d'abord cherché sur GitHub les dépôts Python contenant un AGENTS.md ou un CLAUDE.md à la racine, disposant d'une suite de tests et d'au moins 400 pull requests. Sur cette base, ils ont filtré les PR qui référençaient au moins une issue et modifiaient du code Python. Un agent IA a ensuite trié les PR susceptibles d'introduire des comportements testables de manière déterministe — corrections de bugs ou ajouts de fonctionnalités vérifiables.

Pour chaque PR retenue, un agent a configuré l'environnement d'exécution et vérifié que les tests passaient. Un autre agent a rédigé une description standardisée du ticket en six sections (description, étapes de reproduction, comportement attendu, comportement observé, spécification, informations complémentaires), à partir de la PR et des issues associées. Les chercheurs ont vérifié manuellement que ces descriptions ne divulguaient pas la solution.

La plupart des projets de niche ne fournissent pas de tests unitaires dans leurs PR. Les chercheurs ont donc fait générer des tests par un agent, puis vérifié qu'ils échouaient sur le code d'origine et passaient après application du patch. Ils ont ensuite corrigé manuellement les tests trop spécifiques — ceux qui vérifiaient des détails d'implémentation plutôt que le comportement attendu. Résultat final : 138 instances issues de 12 dépôts, avec une couverture de test moyenne de 75 % du code modifié.

Trois configurations d'évaluation

Chaque instance a été testée dans trois configurations. Sans fichier contextuel : l'agent travaille à l'aveugle, avec seulement le code et la description du ticket. Avec un fichier généré par IA : chaque agent utilise sa propre commande d'initialisation pour produire le fichier (/init pour Codex, par exemple). Avec le fichier rédigé par les développeurs du projet : uniquement sur AgentBench, puisque les projets de SWE-bench n'en possèdent pas.

Quatre agents ont été mis à l'épreuve : Claude Code avec Sonnet 4.5, Codex avec GPT-5.2 et GPT-5.1 Mini, et Qwen Code avec Qwen3-30B. Chaque agent utilisait ses réglages par défaut, température fixée à 0 (sauf Qwen à 0.7), avec un seul échantillon par instance.

Résultats clés

Les fichiers générés par IA font plus de mal que de bien

Sur les 8 combinaisons agent/benchmark testées, les fichiers contextuels générés par IA ont dégradé les performances dans 5 cas. La baisse moyenne atteint 0,5 % sur SWE-bench Lite et 2 % sur AgentBench. En parallèle, le nombre d'étapes par ticket augmente systématiquement — en moyenne 2,5 étapes de plus sur SWE-bench et 3,9 sur AgentBench. Cette exploration supplémentaire se traduit par une hausse du coût d'inférence de 20 à 23 %.

Prenons un exemple concret : avec Claude Code sur SWE-bench Lite, le coût moyen par instance passe de 1,30 $ sans fichier contextuel à 1,51 $ avec un fichier généré. Pour GPT-5.2 sur AgentBench, il bondit de 0,38 $ à 0,57 $ — une augmentation de 50 %.

Les fichiers humains font un peu mieux, mais pas de beaucoup

Les fichiers rédigés par les développeurs surpassent ceux générés par IA dans tous les cas, et améliorent les performances par rapport à l'absence de fichier pour trois agents sur quatre. Le gain moyen reste modeste : +4 % en taux de résolution. Claude Code est l'exception — ses performances baissent même avec un fichier humain. Le surcoût persiste : 3,3 étapes supplémentaires en moyenne, et jusqu'à 19 % de coût en plus.

Les agents suivent les instructions — c'est le problème

L'analyse des traces d'exécution révèle un phénomène éclairant. Quand un fichier contextuel mentionne un outil (par exemple uv pour la gestion des dépendances, ou un outil spécifique au projet), les agents l'utilisent massivement : 1,6 appel par instance en moyenne pour uv quand il est mentionné, contre moins de 0,01 quand il ne l'est pas. Les agents lisent les instructions et les appliquent. Le problème ne vient pas d'un défaut d'obéissance.

Le problème, c'est que ces instructions supplémentaires rendent la tâche plus complexe. GPT-5.2 et GPT-5.1 Mini, dont l'architecture permet de mesurer l'effort de raisonnement via le nombre de tokens de réflexion, consomment respectivement 22 % et 14 % de tokens de raisonnement en plus quand un fichier contextuel est présent. L'agent « réfléchit » davantage, explore plus largement, lance plus de tests — mais ne résout pas mieux le problème.

Les fichiers contextuels ne servent pas de carte du dépôt

L'un des arguments en faveur des fichiers contextuels est qu'ils fournissent une vue d'ensemble du projet. 100 % des fichiers générés par Sonnet 4.5 contiennent une section d'aperçu de l'architecture, 99 % pour GPT-5.2. Les chercheurs ont mesuré le nombre d'étapes avant que l'agent interagisse avec un fichier concerné par le patch. Résultat : la présence d'un fichier contextuel ne réduit pas ce délai. Les agents ne trouvent pas les fichiers pertinents plus vite avec une carte que sans.

Pire, GPT-5.1 Mini perd du temps à chercher le fichier contextuel dans l'arborescence, puis à le relire — alors que son contenu figure déjà dans le contexte initial de l'agent. Ce comportement n'apparaît que lorsqu'un fichier contextuel existe.

Le fichier contextuel duplique la documentation existante

Pour tester cette hypothèse, les chercheurs ont supprimé toute la documentation du dépôt (fichiers .md, dossier docs/, exemples de code) avant l'évaluation. Dans cette configuration sans documentation, les fichiers générés par IA améliorent systématiquement les performances de 2,7 % en moyenne — et surpassent même les fichiers humains. Ce résultat explique les témoignages positifs : les développeurs de projets peu documentés constatent un gain réel en ajoutant un fichier contextuel, parce que celui-ci compense l'absence de documentation, pas parce qu'il apporte une valeur spécifique.

Un meilleur modèle ne produit pas un meilleur fichier

Faire générer les fichiers contextuels par GPT-5.2 (le modèle le plus puissant testé) au lieu du modèle natif de chaque agent améliore les résultats sur SWE-bench Lite (+2 % en moyenne) mais les dégrade sur AgentBench (-3 %). Le choix du prompt d'initialisation — celui de Codex ou celui de Claude Code — n'a pas d'effet cohérent non plus. Aucune combinaison modèle/prompt ne domine les autres de manière fiable.

Implications pratiques

Ces résultats percutent directement les recommandations des fournisseurs d'agents. Anthropic, OpenAI et Alibaba conseillent tous de générer un fichier contextuel pour chaque projet. Les chercheurs suggèrent l'inverse : mieux vaut s'en passer, au moins pour les fichiers auto-générés.

Si un fichier contextuel est maintenu, il devrait se limiter au strict nécessaire. Les données montrent que les agents profitent des informations sur l'outillage spécifique (quel gestionnaire de paquets utiliser, quelle commande lance les tests), mais souffrent des descriptions architecturales détaillées qui les poussent à explorer davantage sans résoudre plus de tickets. Un fichier contextuel efficace ressemblerait à cinq lignes de commandes concrètes, pas à une page de prose sur la philosophie du projet.

Pour les équipes qui gèrent des projets peu documentés ou utilisant des technologies de niche, le calcul change. Dans ces cas, un fichier contextuel — même généré automatiquement — compense l'absence d'information et aide réellement l'agent. La recommandation n'est pas binaire : elle dépend de la qualité de la documentation existante.

Le surcoût financier mérite attention. Une augmentation de 20 % du coût d'inférence, multipliée par des centaines de tickets traités par mois dans une organisation, représente un budget non négligeable — pour un gain en performance nul ou négatif. Les équipes qui mesurent le ROI de leurs agents de code devraient intégrer ce paramètre.

Pour aller plus loin

L'étude se concentre sur Python, un langage abondamment représenté dans les données d'entraînement des modèles. Les agents disposent probablement d'une connaissance paramétrique suffisante de l'écosystème Python pour se passer de fichiers contextuels. En Rust, en Zig ou en Elixir, la donne pourrait être différente. Les auteurs appellent explicitement à des évaluations sur des langages moins courants, où les modèles manquent de connaissances implicites sur les outils et les conventions.

Le benchmark AgentBench ne couvre que la résolution de tickets — corrections de bugs et ajouts de fonctionnalités. Les fichiers contextuels pourraient avoir un impact sur d'autres dimensions : sécurité du code généré, performance, respect des conventions de style. Des travaux antérieurs montrent que le simple fait de demander à un LLM de « générer du code sécurisé » améliore la sécurité du résultat. Un fichier contextuel contenant des consignes de sécurité spécifiques au projet pourrait produire un effet similaire, mais cela reste à mesurer.

La piste la plus prometteuse concerne l'amélioration de la génération automatique elle-même. Les fichiers humains font mieux que les fichiers auto-générés — il existe donc une marge de progression. Des approches de planification et d'apprentissage continu, où l'agent enrichit progressivement sa connaissance d'un dépôt au fil des tâches résolues, pourraient produire des fichiers contextuels adaptatifs plutôt que statiques. Un agent qui se souvient de ses erreurs passées sur un projet donné et met à jour ses propres instructions en conséquence — voilà un scénario où le fichier contextuel deviendrait un véritable outil d'auto-amélioration plutôt qu'une simple description figée.

En attendant, le message est clair : dans l'état actuel des agents et des modèles, un fichier contextuel doit rester court, factuel et orienté vers l'outillage. Les longues descriptions architecturales, les guides de style détaillés et les aperçus exhaustifs de l'arborescence du projet ne font qu'alourdir le travail de l'agent. Moins de prose, plus de commandes.
