Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?

Vulgarisation

Introduction

Plus de 60 000 dépôts GitHub publics contiennent un fichier AGENTS.md ou CLAUDE.md. Ces fichiers ne s'adressent pas aux développeurs humains : ils sont écrits pour les agents de code IA. L'idée semble raisonnable — si un nouveau développeur a besoin d'un guide pour s'orienter dans un projet, un agent autonome devrait en bénéficier aussi. Anthropic, OpenAI, Alibaba : tous recommandent cette pratique. Certains intègrent une commande /init pour générer automatiquement le fichier.

Mais personne n'avait vérifié si ces fichiers servaient à quelque chose. Une équipe de l'ETH Zurich et de LogicStar.ai a posé la question frontalement, avec un protocole expérimental rigoureux couvrant quatre agents de code, deux benchmarks et trois configurations. Leur réponse, publiée en février 2026, prend à contre-pied les recommandations officielles : les fichiers contextuels générés par IA dégradent les performances dans la majorité des cas, et même les fichiers rédigés par des humains n'apportent qu'un gain marginal — pour un surcoût d'inférence supérieur à 20 %.

Le problème

Un agent de code — Claude Code, Codex, Qwen Code — fonctionne en boucle autonome. Il reçoit un ticket, explore le dépôt, modifie des fichiers, lance des tests, corrige ses erreurs, et recommence jusqu'à produire un patch valide. Pour l'aider à s'orienter, on peut lui fournir un fichier contextuel : une description de l'architecture, les conventions de style, les commandes de build, la structure des répertoires.

Deux obstacles rendaient l'évaluation de ces fichiers difficile. Les benchmarks existants comme SWE-bench reposent sur des projets populaires — Django, Matplotlib, scikit-learn — qui ne contiennent pas de fichiers contextuels. Ces fichiers n'ont été formalisés qu'en août 2025. Et les dépôts qui en possèdent sont des projets récents, de niche, avec des pratiques de documentation hétérogènes et des pull requests souvent lacunaires.

Les seules données disponibles se limitaient à des retours anecdotiques : « mon agent fonctionne mieux depuis que j'ai ajouté un AGENTS.md ». Aucune mesure contrôlée, aucun groupe témoin, aucune comparaison systématique entre agents. Les chercheurs devaient construire un nouveau benchmark adapté aux projets de niche, tout en conservant le benchmark de référence pour assurer la comparabilité.

La méthode

L'équipe a travaillé sur deux fronts. Le premier : tester des fichiers contextuels générés par IA sur SWE-bench Lite, un benchmark classique de 300 tickets issus de 11 projets Python majeurs — des projets sans fichier contextuel, donc en évaluant uniquement l'apport des fichiers auto-générés. Le second : créer un benchmark entièrement nouveau, AgentBench, à partir de projets qui utilisent déjà des fichiers contextuels rédigés par leurs développeurs.

La construction d'AgentBench

Imaginez un entonnoir. En haut, les chercheurs ont versé tous les dépôts GitHub Python contenant un AGENTS.md ou un CLAUDE.md à la racine, disposant d'une suite de tests et d'au moins 400 pull requests. Ils ont filtré les PR qui référençaient une issue et modifiaient du code Python. Un agent IA a trié celles qui introduisaient des comportements testables de manière déterministe — corrections de bugs ou ajouts de fonctionnalités vérifiables.

Pour chaque PR retenue, un agent a configuré l'environnement d'exécution et vérifié que les tests passaient. Un second agent a rédigé une description standardisée du ticket en six sections (description, étapes de reproduction, comportement attendu, comportement observé, spécification, informations complémentaires). Les chercheurs ont vérifié manuellement qu'aucune description ne divulguait la solution — sur un échantillon de 10 %, aucune fuite n'a été détectée.

Les projets de niche fournissent rarement des tests unitaires dans leurs PR. Les chercheurs ont fait générer des tests par un agent, puis vérifié qu'ils échouaient sur le code d'origine et passaient après application du patch. Ils ont corrigé manuellement les tests trop spécifiques, ceux qui vérifiaient des détails d'implémentation plutôt que le comportement attendu. Au bout de l'entonnoir : 138 instances issues de 12 dépôts (parmi lesquels smolagents, transformers, fastmcp, tinygrad), avec une couverture de test moyenne de 75 %.

Trois configurations, quatre agents

Chaque instance a été testée dans trois configurations. D'abord sans fichier contextuel : l'agent travaille avec le code et la description du ticket, rien d'autre. Ensuite avec un fichier généré par IA : chaque agent utilise sa propre commande d'initialisation. Enfin avec le fichier écrit par les développeurs du projet — cette troisième configuration n'est disponible que sur AgentBench, puisque les projets de SWE-bench n'en possèdent pas.

Quatre agents ont été mis à l'épreuve : Claude Code avec Sonnet 4.5, Codex avec GPT-5.2 et GPT-5.1 Mini, et Qwen Code avec Qwen3-30B. Température fixée à 0 pour la plupart (0.7 pour Qwen), un seul échantillon par instance, réglages par défaut pour chaque harness.

Résultats clés

Les fichiers générés par IA dégradent les performances

Sur les 8 combinaisons agent/benchmark testées, les fichiers contextuels générés par IA ont réduit le taux de résolution dans 5 cas. La baisse moyenne atteint 0,5 % sur SWE-bench Lite et 2 % sur AgentBench. En parallèle, le nombre d'étapes par ticket augmente systématiquement — 2,5 étapes supplémentaires en moyenne sur SWE-bench, 3,9 sur AgentBench. Cette exploration supplémentaire se traduit par un coût d'inférence en hausse de 20 à 23 %.

Un exemple concret : avec Claude Code sur SWE-bench Lite, le coût moyen par instance passe de 1,30 $ sans fichier contextuel à 1,51 $ avec un fichier généré. Pour GPT-5.2 sur AgentBench, il bondit de 0,38 $ à 0,57 $ — une augmentation de 50 %.

Les fichiers humains aident, mais à la marge

Les fichiers rédigés par les développeurs surpassent ceux générés par IA pour les quatre agents testés, et améliorent les performances par rapport à l'absence de fichier pour trois agents sur quatre. Le gain moyen reste modeste : +4 % en taux de résolution. Claude Code est l'exception — ses performances baissent même avec un fichier humain. Le surcoût persiste : 3,3 étapes supplémentaires et jusqu'à 19 % de coût en plus.

Les agents obéissent — et c'est le problème

L'analyse des traces d'exécution révèle un mécanisme contre-intuitif. Quand un fichier contextuel mentionne un outil — uv pour la gestion des dépendances, ou un outil spécifique au projet — les agents l'utilisent massivement : 1,6 appel par instance en moyenne pour uv quand il est mentionné, contre moins de 0,01 quand il ne l'est pas. Les agents lisent les instructions et les appliquent scrupuleusement.

Mais ces instructions supplémentaires rendent la tâche plus complexe. GPT-5.2 et GPT-5.1 Mini permettent de mesurer l'effort de raisonnement via le nombre de tokens de réflexion adaptatifs. Avec un fichier contextuel, GPT-5.2 consomme 22 % de tokens de raisonnement en plus, GPT-5.1 Mini 14 %. L'agent « réfléchit » davantage, explore plus largement, lance plus de tests — sans résoudre plus de tickets. Les exigences inutiles du fichier contextuel absorbent de la capacité cognitive qui serait mieux employée sur le problème lui-même.

La carte du dépôt ne sert à rien

Un des arguments en faveur des fichiers contextuels : ils fourniraient une vue d'ensemble du projet, accélérant la navigation. 100 % des fichiers générés par Sonnet 4.5 contiennent une section d'aperçu de l'architecture. 99 % pour GPT-5.2. Les chercheurs ont mesuré le nombre d'étapes avant que l'agent touche un fichier concerné par le patch. Résultat : la présence d'un fichier contextuel ne réduit pas ce délai. Les agents ne trouvent pas les fichiers pertinents plus vite avec une carte que sans.

GPT-5.1 Mini perd même du temps supplémentaire à chercher le fichier contextuel dans l'arborescence, puis à le relire — alors que son contenu est déjà injecté dans le contexte initial de l'agent. Ce comportement pathologique n'apparaît que lorsqu'un fichier contextuel existe dans le dépôt.

La redondance comme explication

Les chercheurs ont testé une hypothèse : les fichiers auto-générés dupliquent l'information déjà présente dans la documentation du projet. Pour le vérifier, ils ont supprimé toute la documentation (fichiers .md, dossier docs/, exemples de code) avant l'évaluation. Dans cette configuration dépouillée, les fichiers générés par IA améliorent systématiquement les performances — +2,7 % en moyenne — et surpassent même les fichiers humains.

Ce résultat éclaire les témoignages positifs des développeurs individuels. Les projets peu documentés bénéficient réellement d'un fichier contextuel, parce qu'il compense l'absence de documentation. Ce n'est pas le fichier contextuel qui apporte de la valeur : c'est la documentation, sous quelque forme qu'elle prenne.

Ni le modèle ni le prompt ne changent la donne

Faire générer les fichiers par GPT-5.2, le modèle le plus puissant testé, au lieu du modèle natif de chaque agent, améliore les résultats sur SWE-bench Lite (+2 %) mais les dégrade sur AgentBench (-3 %). Utiliser le prompt d'initialisation de Codex plutôt que celui de Claude Code n'a pas d'effet cohérent non plus. Aucune combinaison modèle/prompt ne domine de manière fiable. Le problème ne se résout pas en améliorant la qualité de la génération.

Implications pratiques

Ces résultats percutent directement les recommandations des fournisseurs d'agents. Anthropic, OpenAI et Alibaba conseillent tous de générer un fichier contextuel pour chaque projet. Les données disent l'inverse : pour un projet Python correctement documenté, mieux vaut s'en passer.

Si un fichier contextuel est maintenu, il devrait se limiter au strict minimum. Les agents profitent des informations sur l'outillage spécifique — quel gestionnaire de paquets utiliser, quelle commande lance les tests — mais souffrent des descriptions architecturales détaillées qui les poussent à explorer sans résoudre plus de problèmes. Un fichier contextuel efficace ressemblerait à cinq lignes de commandes concrètes, pas à une page de prose sur la philosophie du projet.

Le surcoût financier mérite calcul. Une augmentation de 20 % du coût d'inférence, multipliée par des centaines de tickets traités par mois, représente un budget conséquent — pour un gain en performance nul ou négatif. Les équipes qui mesurent le retour sur investissement de leurs agents de code devraient intégrer ce paramètre dans leur évaluation.

Pour les équipes qui gèrent des projets peu documentés ou utilisant des technologies de niche, la conclusion diffère. Un fichier contextuel — même généré automatiquement — compense l'absence d'information et aide l'agent. La recommandation dépend du niveau de documentation existant : si le projet possède déjà un README détaillé, un dossier docs/ et des exemples de code, le fichier contextuel n'apportera rien. S'il n'en possède pas, il comblera un vide réel.

Un enseignement sous-jacent traverse toute l'étude : les agents de code actuels ne manquent pas de docilité. Ils suivent les instructions qu'on leur donne. Le vrai défi est de leur donner les bonnes instructions — celles qui aident sans encombrer. Chaque directive inutile dans un fichier contextuel consomme de la capacité de raisonnement et allonge la résolution. Le minimalisme est une vertu opérationnelle.

Pour aller plus loin

L'étude se concentre sur Python, un langage abondamment représenté dans les données d'entraînement des modèles. Les agents disposent probablement d'une connaissance implicite suffisante de l'écosystème Python — ses outils, ses conventions, ses frameworks — pour se passer de fichiers contextuels. En Rust, en Zig ou en Elixir, les modèles manquent de cette familiarité. Un fichier contextuel décrivant la chaîne de compilation, les conventions de nommage ou les outils de test spécifiques pourrait avoir un impact bien plus marqué. Les auteurs appellent à des évaluations sur ces langages moins représentés.

Le benchmark AgentBench ne mesure que la résolution de tickets — corrections de bugs et ajouts de fonctionnalités. Les fichiers contextuels pourraient influencer d'autres dimensions du travail d'un agent : la sécurité du code produit, ses performances, le respect des conventions de style. Des travaux antérieurs montrent que le simple fait de demander à un LLM de produire du code sécurisé améliore la sécurité du résultat. Un fichier contextuel contenant des consignes de sécurité spécifiques au projet pourrait produire un effet similaire — mais cela reste à démontrer.

La piste la plus stimulante concerne l'amélioration de la génération automatique elle-même. Les fichiers humains font mieux que les fichiers auto-générés : il existe donc une marge de progression. Des approches d'apprentissage continu, où l'agent enrichit progressivement sa connaissance d'un dépôt au fil des tâches résolues, pourraient produire des fichiers contextuels adaptatifs plutôt que statiques. Un agent qui se souvient de ses erreurs passées sur un projet donné et met à jour ses propres instructions en conséquence transformerait le fichier contextuel en outil d'auto-amélioration — au lieu d'une description figée rédigée une fois pour toutes.

Le message de fond est limpide : dans l'état actuel des modèles et des agents, un fichier contextuel doit rester court, factuel et orienté vers l'outillage. Les descriptions architecturales détaillées, les guides de style exhaustifs et les aperçus d'arborescence alourdissent le travail de l'agent sans l'accélérer. La prochaine génération d'agents de code devra apprendre à extraire du contexte ce qui est utile et à ignorer le reste — ou mieux, à construire elle-même le contexte dont elle a besoin, tâche par tâche.
