SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks

Li, X. et al. — arXiv preprint, février 2026

1. Objectif et contexte

Les Agent Skills — paquets structurés de connaissances procédurales (instructions, templates de code, ressources exécutables) — augmentent les agents LLM au moment de l'inférence sans modification du modèle. Malgré une adoption explosive (47 150 Skills uniques recensées dans l'écosystème), aucun benchmark n'évaluait systématiquement si et quand ces Skills améliorent réellement les performances des agents.

SKILLSBENCH comble cette lacune en proposant le premier benchmark traitant les Skills comme des artefacts d'évaluation de premier ordre. L'objectif est triple :

  • Quantifier l'efficacité des Skills curées vs. auto-générées
  • Identifier les facteurs de design (quantité, complexité) qui déterminent l'efficacité
  • Comparer l'impact des Skills selon les domaines, modèles et harnesses agents

La question de recherche est claire, originale et d'une pertinence immédiate pour les praticiens et chercheurs du domaine. L'analogie architecturale proposée (modèle = CPU, harness = OS, Skills = applications) fournit un cadre conceptuel utile.

2. Méthodologie

2.1 Construction du benchmark

Le dataset comprend 84 tâches évaluées (sur 86 curées, issues de 322 soumissions candidates) couvrant 11 domaines : Software Engineering (16), Office & White Collar (14), Natural Science (12), Media & Content Production (11), Cybersecurity (8), Finance (8), Robotics (5), Manufacturing (3), Energy (3), Mathematics (2), Healthcare (2).

Chaque tâche est un module autonome comprenant :

  • Une instruction humaine (vérifiée par GPTZero et revue humaine)
  • Un environnement Docker containerisé avec Skills
  • Une solution oracle (100% de réussite requise)
  • Des vérificateurs déterministes (pytest, pas de LLM-as-judge)

La construction suit un modèle communautaire open-source (105 contributeurs) avec un pipeline de qualité en trois phases : validation automatisée, revue humaine, et audit de fuite.

2.2 Protocole expérimental

Trois conditions évaluées par tâche :

  • No Skills : instruction seule
  • With Skills : Skills curées complètes
  • Self-Generated Skills : l'agent génère ses propres connaissances procédurales avant de résoudre

7 configurations agent-modèle testées : Claude Code (Opus 4.5, Opus 4.6, Sonnet 4.5, Haiku 4.5), Gemini CLI (Gemini 3 Pro, Gemini 3 Flash), Codex CLI (GPT-5.2). Température fixée à 0, 5 essais par tâche, totalisant 7 308 trajectoires valides.

Les métriques principales sont le pass rate (binaire, agrégé par tâche puis par configuration) et le normalized gain (formule de Hake), permettant des comparaisons entre modèles à baselines différentes.

2.3 Forces méthodologiques

  • Vérification déterministe éliminant la variance du LLM-as-judge
  • Évaluation appariée (with/without Skills) sur les mêmes tâches
  • Audit de fuite automatisé pour empêcher les Skills de contenir des solutions directes
  • Stratification par difficulté (Core/Extended/Extreme)
  • Taux d'acceptation de 26.7% reflétant un filtrage rigoureux

2.4 Faiblesses méthodologiques

  • Absence de baselines contrôlées en longueur : l'injection de Skills augmente la longueur du contexte, rendant impossible l'attribution causale stricte entre « plus de contexte » et « connaissances procédurales structurées »
  • Biais d'entraînement potentiel : les modèles Claude ont été entraînés avec connaissance de la spécification Agent Skills, conférant un avantage potentiel
  • Déséquilibre des domaines : Healthcare (2 tâches) et Manufacturing (3) sont sous-représentés, fragilisant les conclusions spectaculaires (+51.9pp, +41.9pp) pour ces domaines
  • Écart benchmark-écosystème : les Skills évaluées proviennent du quartile supérieur en qualité (score ≥9/12), alors que l'écosystème réel a un score moyen de 6.2/12

3. Résultats principaux

Finding 1 : Les Skills curées apportent un bénéfice substantiel mais variable

Amélioration moyenne de +16.2 points de pourcentage (pp) avec une forte variance (de +13.6pp à +23.3pp selon la configuration). Claude Code + Opus 4.5 montre le gain maximal (+23.3pp), reflétant l'intégration native des Skills.

Finding 2 : Les Skills auto-générées sont inefficaces

Résultat négatif important : les modèles ne peuvent pas produire de manière fiable les connaissances procédurales dont ils bénéficient en consommation. Moyenne de –1.3pp par rapport à la baseline sans Skills. Seul Opus 4.6 montre une amélioration modeste (+1.4pp). Deux modes d'échec identifiés : procédures imprécises et non-reconnaissance du besoin de Skills spécialisées.

Finding 3 : Forte hétérogénéité par domaine

Healthcare (+51.9pp) et Manufacturing (+41.9pp) bénéficient le plus, tandis que Mathematics (+6.0pp) et Software Engineering (+4.5pp) montrent des gains faibles. Interprétation cohérente : les domaines sous-représentés dans le pré-entraînement bénéficient davantage de la guidance procédurale externe.

Finding 4 : « Less is more » pour le design de Skills

2-3 Skills sont optimales (+18.6pp) ; 4+ Skills n'apportent que +5.9pp. Les Skills « Detailed » (+18.8pp) et « Compact » (+17.1pp) surpassent les Skills « Comprehensive » (–2.9pp). La surcharge documentaire dégrade les performances.

Finding 5 : Les Skills compensent partiellement l'échelle du modèle

Claude Haiku 4.5 avec Skills (27.7%) dépasse Claude Opus 4.5 sans Skills (22.0%), démontrant que les Skills peuvent substituer partiellement la capacité du modèle.

Finding 6 : 16 tâches sur 84 montrent des deltas négatifs

Les Skills peuvent nuire lorsqu'elles introduisent une guidance conflictuelle ou une complexité inutile pour des tâches que le modèle maîtrise déjà.

4. Discussion et limites

Limites reconnues par les auteurs

  • Couverture : tâches terminales containerisées uniquement — les résultats ne se transfèrent pas directement aux agents GUI, multi-agents ou workflows de long horizon
  • Attribution causale : l'augmentation du contexte est un confondant non contrôlé ; des baselines à longueur égale (texte aléatoire, documentation RAG) sont nécessaires
  • Déterminisme : la containerisation fournit l'isolation mais pas le déterminisme parfait ni l'immunité à la contamination des données d'entraînement
  • Volatilité des harnesses : les harnesses commerciales évoluent, limitant la reproductibilité à long terme
  • Composition de Skills : les effets d'interaction entre Skills multiples ne sont pas étudiés

Limites additionnelles identifiées

  • Biais de sélection des Skills : en ne retenant que le quartile supérieur de qualité, le benchmark surestime l'efficacité des Skills en conditions réelles
  • Petit nombre de tâches par domaine : 2 tâches pour Healthcare rendent le +51.9pp statistiquement fragile — un intervalle de confiance aurait été bienvenu
  • Absence d'analyse statistique formelle : pas de tests de significativité, pas d'intervalles de confiance, pas de correction pour comparaisons multiples
  • Biais de l'évaluation binaire : le pass/fail ne capture pas les progrès partiels ni la qualité des solutions

5. Reproductibilité

Points positifs :

  • Environnements Docker containerisés assurant l'isolation
  • Vérificateurs déterministes (pytest, pas de LLM-as-judge)
  • Infrastructure open-source mentionnée (framework Harbor)
  • Appendices détaillés avec spécifications complètes des tâches, structure des répertoires, templates de configuration
  • 5 essais par tâche pour lisser la variance

Points faibles :

  • Dépendance à des APIs commerciales (Claude Code, Gemini CLI, Codex CLI) susceptibles d'évoluer
  • Les Skills curées ne sont pas toutes détaillées dans l'article
  • Le code du benchmark et les données ne sont pas explicitement publiés avec un lien dans cet article (préprint)
  • Température 0 ne garantit pas le déterminisme parfait pour tous les modèles

6. Impact et applications

Impact scientifique :

  • Premier benchmark dédié à l'évaluation systématique des Agent Skills — comble un vide méthodologique important
  • Le résultat sur les Skills auto-générées est particulièrement impactant : il démontre une asymétrie fondamentale entre production et consommation de connaissances procédurales par les LLM
  • Les principes de design empiriquement validés (concision, modularité, 2-3 modules) fournissent des recommandations actionnables

Impact pratique :

  • Guide le design de Skills : préférer des instructions focalisées avec exemples concrets plutôt que des documentations exhaustives
  • Justifie l'investissement dans la curation humaine de Skills plutôt que la génération automatique
  • Démontre le potentiel des Skills pour démocratiser l'accès aux capacités agents (petits modèles + Skills ≈ grands modèles)
  • Alerte sur les risques de dégradation (16/84 tâches négativement impactées)

7. Positionnement dans la littérature

SKILLSBENCH se positionne à l'intersection de trois courants de recherche :

Benchmarks agents : contrairement à Terminal-Bench (Merrill et al., 2026), SWE-bench (Jimenez et al., 2024), AgentBench (Liu et al., 2023) et WebArena (Zhou et al., 2024) qui mesurent la capacité brute des modèles, SKILLSBENCH introduit l'évaluation appariée de l'augmentation — mesurant non pas « que peut faire le modèle ? » mais « combien l'augmentation par Skills améliore-t-elle le modèle ? ».

Augmentation procédurale : s'inscrit dans la lignée de CoALA (Sumers et al., 2023), Voyager (Wang et al., 2023), et des travaux sur les options temporelles (Sutton et al., 1999), en opérationnalisant le concept de connaissances procédurales modulaires pour les agents LLM.

Écosystèmes de Skills : documente quantitativement pour la première fois l'écosystème de 47 150 Skills (croissance exponentielle, distribution de qualité, caractéristiques structurelles), fournissant un contexte empirique aux spécifications d'Anthropic (Agent Skills, MCP).

La contribution principale réside dans le passage d'une évaluation agent-centrique à une évaluation augmentation-centrique, un changement de paradigme nécessaire étant donné la prolifération des écosystèmes de Skills.
