SkillsBench: Benchmarking How Well Agent Skills Work Across Diverse Tasks

Lecture Critique d'Article

Objectif et contexte

Les Agent Skills — paquets structurés d'instructions procédurales, de scripts et de modèles de code — se multiplient dans l'écosystème des agents LLM depuis l'automne 2025. Plus de 47 000 Skills uniques circulent sur GitHub et des marketplaces communautaires. Aucune évaluation systématique ne mesurait leur apport réel sur la performance des agents avant SkillsBench.

La question de recherche est triple : dans quelle mesure les Skills améliorent-elles la performance des agents ? Quels facteurs de conception distinguent les Skills efficaces des inefficaces ? Les modèles peuvent-ils générer eux-mêmes des Skills utiles ? Le benchmark couvre 84 tâches réparties sur 11 domaines (ingénierie logicielle, finance, santé, cybersécurité, sciences naturelles, robotique, fabrication, énergie, médias, bureautique, mathématiques), évaluées sous trois conditions sur 7 configurations agent-modèle, pour un total de 7 308 trajectoires.

Méthodologie

Le protocole repose sur une évaluation appariée : chaque tâche est exécutée avec et sans Skills, ce qui permet de mesurer directement le delta d'efficacité. Trois harnais commerciaux sont mobilisés : Claude Code (Anthropic), Gemini CLI (Google) et Codex CLI (OpenAI), couplés à sept modèles frontière — Claude Opus 4.5, Opus 4.6, Sonnet 4.5, Haiku 4.5, Gemini 3 Pro, Gemini 3 Flash et GPT-5.2.

Les tâches s'exécutent dans des conteneurs Docker isolés. La vérification est déterministe via des assertions pytest, sans recours au LLM-as-judge. Chaque trajectoire produit un résultat binaire pass/fail. Cinq essais par tâche et par condition alimentent le calcul d'un taux de réussite moyen, avec intervalles de confiance bootstrap à 95 % sur 1 000 rééchantillonnages.

La construction du jeu de données suit un modèle communautaire : 105 contributeurs ont soumis 322 tâches candidates. Le filtrage combine validation automatisée (structure, oracle à 100 %, détection IA via GPTZero) et revue humaine (validité des données, réalisme, qualité de l'oracle, qualité des Skills, résistance à la triche). Le taux d'acceptation atteint 26,7 %. Un audit de fuite automatisé, exécuté via un agent Claude Code SDK en CI, vérifie que les Skills ne contiennent pas de solutions directes.

Un biais de sélection assumé : les Skills curées proviennent du quartile supérieur en qualité (score ≥ 9/12), alors que la moyenne de l'écosystème est de 6,2/12. Les résultats représentent donc un scénario optimiste.

Résultats principaux

Les Skills curées augmentent le taux de réussite moyen de 16,2 points de pourcentage (pp), de 24,3 % à 40,6 %. L'amplitude varie selon la configuration : +13,6 pp pour Gemini CLI + Gemini 3 Pro, +23,3 pp pour Claude Code + Opus 4.5. La meilleure performance absolue revient à Gemini CLI + Gemini 3 Flash (48,7 % avec Skills).

Les écarts entre domaines sont considérables. La santé progresse de +51,9 pp et la fabrication de +41,9 pp, tandis que les mathématiques gagnent +6,0 pp et l'ingénierie logicielle +4,5 pp. Les domaines dont les connaissances procédurales sont sous-représentées dans le pré-entraînement profitent le plus de l'augmentation externe.

Le résultat le plus frappant concerne les Skills auto-générées : –1,3 pp en moyenne par rapport à la condition sans Skills. Seul Opus 4.6 montre un gain marginal (+1,4 pp) ; Codex + GPT-5.2 se dégrade de –5,6 pp. Les modèles ne parviennent pas à produire les connaissances procédurales dont ils bénéficient en consommation. L'analyse des trajectoires révèle deux modes de défaillance : production de procédures imprécises (« utiliser pandas pour le traitement de données » sans préciser les patterns API concrets), ou incapacité à reconnaître le besoin de Skills spécialisées dans les domaines à forte expertise (fabrication, finance).

L'analyse par tâche révèle 16 cas sur 84 où les Skills dégradent la performance, jusqu'à –39,3 pp pour taxonomy-tree-merge. Les Skills peuvent introduire des conflits de guidage ou une complexité superflue sur des tâches que les modèles gèrent déjà sans aide.

Sur la conception des Skills : 2 à 3 modules ciblés surpassent les documentations exhaustives. Les Skills « détaillées » (+18,8 pp) et « compactes » (+17,1 pp) dominent les « compréhensives » (–2,9 pp). La concision procédurale l'emporte sur l'exhaustivité.

Claude Haiku 4.5 avec Skills (27,7 %) dépasse Opus 4.5 sans Skills (22,0 %). Les Skills compensent partiellement les limites de capacité du modèle, avec des implications économiques directes : Gemini 3 Flash avec Skills coûte 0,57 $ par essai contre 1,06 $ pour Gemini 3 Pro, tout en offrant une meilleure performance.

Discussion et limites

Le principal confondant méthodologique est l'augmentation de la longueur du contexte. L'injection de Skills ajoute du texte au prompt, et une partie du gain pourrait refléter un effet « plus de contexte » plutôt que la structure procédurale. Les auteurs argumentent que la condition auto-générée (même budget, pas de gain) plaide en faveur de la structure. L'argument tient partiellement, mais un baseline avec du texte aléatoire ou de la documentation factuelle de longueur équivalente aurait fourni une isolation causale bien plus solide. Les auteurs identifient eux-mêmes cette lacune et la listent comme travail futur.

La sélection des Skills dans le quartile supérieur de qualité crée un écart entre le benchmark et l'usage réel. Un praticien piochant dans l'écosystème moyen (6,2/12) obtiendra probablement des gains inférieurs aux 16,2 pp rapportés.

La couverture par domaine est inégale. La santé et les mathématiques comptent 2 tâches chacune, la fabrication et l'énergie 3. Le gain spectaculaire de +51,9 pp en santé repose sur 2 tâches seulement — impossible d'en tirer une conclusion généralisable. Les domaines les mieux représentés (ingénierie logicielle : 16 tâches, bureautique : 14) offrent des estimations plus fiables.

Le benchmark se limite aux tâches en terminal conteneurisé. Les agents GUI, les systèmes multi-agents et les workflows à horizon long ne sont pas couverts. La métrique binaire pass/fail écrase l'information sur les progrès partiels : un agent qui résout 90 % d'une tâche et un autre qui n'a rien produit reçoivent le même score de 0. L'analyse des échecs (Appendice I) compense partiellement cette perte avec une taxonomie des modes de défaillance.

Le recours à des harnais commerciaux introduit un risque de non-reproductibilité à moyen terme. Les versions sont documentées (Claude Code 2.1.19, modèles identifiés par leur API ID), mais le comportement des harnais évolue entre versions. Le couplage étroit modèle-harnais (Claude Code ne fonctionne qu'avec les modèles Claude, etc.) empêche d'isoler les effets du modèle et du harnais.

Reproductibilité

Les 84 tâches sont conteneurisées avec des Dockerfiles complets, des Makefiles de build et des budgets de ressources spécifiés dans des fichiers task.toml. L'environnement d'exécution est reproductible à l'identique. Les identifiants de modèle et les versions des harnais figurent en Table 8. La température est fixée à 0.

La dépendance à des API commerciales conditionne la reproduction exacte à la disponibilité et au coût de ces services. Aucun lien vers un dépôt public n'est fourni dans le papier, bien que l'infrastructure soit décrite comme ouverte. La validation à 10 essais sur un sous-ensemble (tâches Extreme, GPT-5.2 et Opus 4.5) confirme la stabilité des estimations à 5 essais, avec un delta ≤ 1,2 pp.

Le processus de construction du benchmark est documenté avec un degré de détail inhabituel : pipeline CI automatisé, critères de revue explicites, template de rapport de benchmark, cycle de vie des pull requests. Cette transparence procédurale dépasse la norme des publications de benchmarks dans le domaine.

Impact et applications

SkillsBench produit les premières données empiriques à grande échelle sur l'efficacité des Agent Skills. Les praticiens disposent d'un cadre d'évaluation pour décider si l'adoption de Skills se justifie dans leur domaine — les gains de +51,9 pp en santé contre +4,5 pp en ingénierie logicielle orientent les priorités d'investissement, sous réserve des caveats de taille d'échantillon mentionnés plus haut.

Les principes de conception validés empiriquement — Skills focalisés de 2 à 3 modules, instructions procédurales concises plutôt que documentation exhaustive, au moins un exemple fonctionnel — fournissent un guide opérationnel pour les auteurs de Skills. La démonstration que les Skills auto-générées sont inefficaces recentre les efforts sur la curation humaine experte, un résultat contre-intuitif dans un domaine qui tend vers l'automatisation maximale.

L'analyse coût-performance montre que les Skills décalent la frontière de Pareto vers le haut. Un petit modèle augmenté de Skills peut rivaliser avec un modèle plus puissant sans Skills, à moindre coût. Ce résultat a des implications directes pour les architectures de déploiement en production.

Positionnement dans la littérature

SkillsBench se distingue des benchmarks agents existants — Terminal-Bench (Merrill et al., 2026), SWE-bench (Jimenez et al., 2024), AgentBench (Liu et al., 2023), WebArena (Zhou et al., 2024) — par son évaluation appariée. Ces benchmarks mesurent la capacité brute d'un agent sur une tâche. SkillsBench mesure le gain d'augmentation, une question complémentaire et jusqu'ici non traitée.

Le concept de Skills hérite des travaux sur l'abstraction temporelle (options framework, Sutton et al., 1999), les architectures cognitives pour agents linguistiques (CoALA, Sumers et al., 2023), et les agents à monde ouvert (Voyager, Wang et al., 2023). La spécification formelle — fichier SKILL.md avec frontmatter YAML, ressources exécutables optionnelles, portabilité cross-modèle — formalise ce qui existait de manière informelle dans les ecosystèmes Claude Code et Gemini CLI.

La méthodologie combine le scoring à dénominateur fixe de Terminal-Bench et le gain normalisé de Hake (1998), emprunté à la recherche en physique éducative. Cette combinaison permet de comparer les améliorations entre modèles de niveaux de base différents, en distinguant les effets de plafond des effets réels de scaffolding. Malgré les limites discutées — confondant de longueur de contexte, biais de sélection des Skills, couverture inégale des domaines — SkillsBench établit un cadre d'évaluation qui manquait au domaine et qui sera probablement adopté comme référence pour les travaux futurs sur l'augmentation procédurale des agents.
