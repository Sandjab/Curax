On the Impact of AGENTS.md Files on the Efficiency of AI Coding Agents

Vulgarisation

1. Introduction : le mode d'emploi que personne n'avait testé

Soixante mille dépôts GitHub embarquent aujourd'hui un fichier AGENTS.md. Ce fichier ne s'adresse pas aux développeurs humains. Il parle aux agents IA — ces programmes autonomes capables de cloner un dépôt, lire le code, exécuter des commandes et soumettre des pull requests sans intervention humaine. OpenAI Codex, Claude Code, GitHub Copilot Workspace : ces outils ne complètent plus des lignes de code, ils prennent en charge des tâches entières de développement.

Mais une question restait sans réponse empirique : est-ce qu'écrire un fichier d'instructions pour ces agents change réellement leur efficacité ? Ou est-ce du cargo cult — un rituel rassurant sans effet mesurable ?

Une équipe internationale de six chercheurs, rattachés à Singapour, Heidelberg, Bamberg et Londres, a monté une expérience contrôlée pour trancher. Ils ont fait tourner un agent IA sur 124 pull requests réelles, tirées de 10 dépôts open source, dans deux conditions : avec et sans fichier AGENTS.md. Les résultats montrent une réduction de 29 % du temps d'exécution médian et de 17 % de la consommation de tokens de sortie quand le fichier est présent. Le tout sans dégradation visible de la qualité du code produit.

2. Le problème : un développeur largué le premier jour

Quand un nouveau développeur rejoint un projet, il passe ses premiers jours à comprendre l'environnement. Où se trouvent les fichiers de configuration ? Quelle commande lance les tests ? Quelles conventions de nommage l'équipe utilise-t-elle ? Il lit le README, parcourt l'arborescence, interroge ses collègues. Ce temps d'apprentissage est incompressible.

Un agent IA de codage traverse exactement la même phase, mais sans collègues à qui poser des questions. Quand Codex débarque sur un dépôt inconnu, il doit inférer l'architecture en explorant les fichiers un par un. Il lit des dizaines de fichiers sources, tente des commandes, revient en arrière quand il se trompe. Cette exploration coûte du temps et des tokens — la monnaie d'échange des modèles de langage, facturée à l'unité par les fournisseurs d'API.

Le prompt engineering — l'art de bien formuler les requêtes envoyées au modèle — a concentré l'attention des chercheurs ces dernières années. Mais un prompt est éphémère. Il disparaît à la fin de chaque session. Les fichiers AGENTS.md prennent le problème sous un autre angle : ils persistent dans le dépôt, se versionnent avec Git, et fournissent à l'agent un socle de connaissances stable. Architecture du projet, conventions de codage, commandes de build et de test, contraintes techniques — tout ce qu'un humain apprendrait en quelques jours, condensé dans un document que l'agent ingère avant de commencer.

Deux études récentes avaient cartographié le phénomène. Chatlatanagulchai et ses collègues ont analysé le contenu de ces fichiers : les développeurs y inscrivent surtout des instructions fonctionnelles (build, tests, implémentation) et assez peu de directives sur la performance ou la sécurité. Mohsenimofidi et al. ont documenté leur adoption croissante comme mécanisme de « context engineering ». Mais aucune de ces études ne mesurait l'effet concret sur le comportement d'un agent. L'article de Lulla et al. comble cette lacune.

3. La méthode : un test A/B sur du vrai code

Le principe de l'expérience est celui d'un test A/B, transposé au génie logiciel. Pour chaque tâche de développement, l'agent Codex (basé sur le modèle gpt-5.2-codex) est lancé deux fois sur le même dépôt, dans le même état exact du code. Une fois avec le fichier AGENTS.md à la racine, une fois sans. Tout le reste — code source, description de la tâche, environnement Docker — reste identique. Les chercheurs mesurent le temps d'exécution mural et le nombre de tokens consommés dans chaque condition.

Filtrage des dépôts

Les chercheurs sont partis d'un corpus de 132 dépôts identifiés dans des travaux antérieurs. Premier filtre : ne garder que ceux avec un seul AGENTS.md, placé à la racine — pour éviter les interférences entre fichiers d'instructions dispersés dans les sous-dossiers. Ce critère réduit l'ensemble à 89 dépôts. Second filtre : éliminer les fichiers trop superficiels. Un modèle de langage local (gpt-oss-120b, exécuté via Ollama) a classé chaque fichier selon qu'il documentait les conventions de codage, l'architecture du projet ou une description générale. Seuls les fichiers couvrant au moins ces trois dimensions ont été retenus, après vérification manuelle. Résultat : 26 dépôts qualifiés, dont 10 tirés au sort pour l'expérience.

Sélection des tâches

Sur chacun de ces 10 dépôts, les chercheurs ont sélectionné jusqu'à 15 pull requests déjà fusionnées. Critères stricts : moins de 100 lignes modifiées (ajouts + suppressions), 5 fichiers touchés au maximum, uniquement du code (pas de documentation ni de fichiers de configuration), et une date postérieure à l'introduction du fichier AGENTS.md. Ces contraintes donnent des tâches de taille maîtrisée, reproductibles, et représentatives du quotidien d'un développeur : corrections de bugs, petits ajouts fonctionnels, refactorisations ciblées. Au total, 124 pull requests ont été retenues.

Remonter le temps dans Git

Pour chaque PR, le dépôt est ramené à l'état qu'il avait juste avant la fusion. L'agent doit alors reproduire les changements sans connaître la solution. C'est comme donner à un stagiaire le ticket associé à une PR déjà mergée et lui demander de produire le même résultat en partant de zéro. La version du fichier AGENTS.md utilisée est celle qui existait à ce point précis de l'historique.

Des tickets standardisés

Problème pratique : beaucoup de PRs sur GitHub ont des descriptions inutilisables — un titre « fix bug » sans explication. Pour donner à l'agent un input cohérent, les chercheurs ont généré pour chaque PR une description structurée façon ticket GitHub (énoncé du problème, comportement attendu, critères d'acceptation). Un modèle de langage local recevait le diff de la PR et l'arborescence du dépôt, puis produisait cette description normalisée.

Isolation complète

Chaque exécution tourne dans un conteneur Docker dédié. Le dépôt est cloné, positionné sur le bon commit, et l'agent reçoit sa tâche. Après exécution, tout est nettoyé : aucun cache, aucun artefact ne survit d'une tâche à l'autre. Ce protocole garantit que chaque run part d'un état parfaitement identique.

4. Résultats clés : 29 % plus vite, 17 % de tokens en moins

Le temps d'exécution médian passe de 98,6 secondes sans AGENTS.md à 70,3 secondes avec — une baisse de 28,6 %. En moyenne, l'agent gagne 33 secondes par tâche (de 163 s à 130 s). L'alignement entre la réduction de la moyenne (20 %) et celle de la médiane (29 %) indique que le gain n'est pas tiré par quelques cas extrêmes. L'amélioration est distribuée sur l'ensemble des tâches.

La consommation de tokens de sortie — ceux que l'agent génère, et la composante la plus chère de la facturation API — baisse en moyenne de 5 745 à 4 591 tokens (−20 %). La médiane passe de 2 925 à 2 440 (−17 %). La réduction plus forte de la moyenne que de la médiane livre un indice : le fichier AGENTS.md bénéficie surtout aux tâches qui, sans lui, déclenchent des exécutions anormalement gourmandes. L'agent qui tâtonne sur un dépôt mal documenté consomme parfois un volume de tokens disproportionné en exploration ; le fichier d'instructions comprime ces pics.

Les tokens d'entrée (ceux envoyés au modèle) diminuent aussi en moyenne (de 353 000 à 319 000, soit −10 %), mais leur médiane reste stable, voire légèrement supérieure avec AGENTS.md. L'explication tient en une phrase : le fichier d'instructions lui-même ajoute du contexte en entrée, ce qui compense partiellement la réduction obtenue par une navigation moins exploratoire.

Les tests de Wilcoxon (test statistique pour données appariées, adapté quand on compare les mêmes tâches dans deux conditions) confirment la significativité pour le temps d'exécution et les tokens de sortie (p < 0,05). Les différences sur les tokens d'entrée et les tokens en cache ne franchissent pas ce seuil.

Un AGENTS.md ne rend pas l'agent plus intelligent. Il lui évite de perdre du temps à comprendre ce qu'un développeur humain aurait appris en lisant le README et en discutant avec l'équipe.

Les chercheurs ont vérifié que ces gains d'efficacité ne masquent pas une dégradation de la qualité. Sur 50 tâches échantillonnées, l'agent a produit des modifications de code non triviales et cohérentes avec l'objectif de la PR d'origine — avec ou sans fichier d'instructions. Le fichier n'incite pas l'agent à bâcler. Il l'aide à aller droit au but.

5. Implications pratiques : un levier d'optimisation à coût quasi nul

Rédiger un fichier AGENTS.md prend quelques dizaines de minutes à un développeur senior qui connaît le projet. L'effort est minime ; le retour, mesurable. Prenons un cas concret : une équipe qui exécute 500 tâches automatisées par semaine via un agent type Codex. Une réduction de 20 % des tokens de sortie allège directement la facture d'API. Les 29 % gagnés sur le temps médian libèrent de la capacité d'exécution — ou raccourcissent les files d'attente quand plusieurs agents tournent en parallèle sur un même serveur CI.

Le ratio investissement/bénéfice est difficile à battre. Optimiser un modèle de langage demande des mois de travail et des millions en calcul. Régler l'architecture d'un agent nécessite des compétences pointues en ingénierie logicielle. Un fichier AGENTS.md, c'est un document texte que n'importe quel développeur peut écrire, versionner, relire en code review et faire évoluer avec le projet. Du context engineering à l'état brut.

L'étude met aussi en lumière un dédoublement dans la documentation des dépôts. Un README s'adresse aux humains : il explique le « pourquoi » du projet, son périmètre, ses dépendances. Un AGENTS.md s'adresse aux machines : il encode le « comment » opérationnel — quelles commandes lancer, quelles conventions respecter, quelle structure de dossiers suivre. Les deux coexistent sans se substituer l'un à l'autre. Cette distinction va s'accentuer à mesure que les agents gagnent en autonomie et interviennent sur des projets de plus en plus variés.

Un risque à anticiper : la maintenance. Un AGENTS.md obsolète, décrivant une architecture qui n'existe plus ou des commandes de build périmées, pourrait induire l'agent en erreur et produire l'effet inverse. L'étude ne mesure pas ce scénario, mais le bon sens le suggère. Traiter ce fichier comme du code — le mettre à jour à chaque changement d'architecture, surveiller les résultats de l'agent comme indicateur indirect de sa pertinence — relève d'une hygiène élémentaire de projet.

Un point souvent négligé : la transparence. Contrairement à un prompt système caché dans une interface, un fichier AGENTS.md est visible par toute l'équipe. Il peut être relu, commenté, amendé en pull request. Cette transparence crée un canal de communication explicite entre les développeurs et les agents qui opèrent sur leur code — un canal qui n'existait pas quand les instructions restaient enfouies dans des prompts éphémères.

6. Pour aller plus loin : ce que l'étude ne dit pas encore

Les auteurs délimitent clairement les frontières de leur travail.

L'expérience porte sur un seul agent (Codex) et un seul modèle (gpt-5.2-codex). Claude Code, Copilot Workspace, Devin et les autres systèmes interprètent les fichiers d'instructions différemment. Chaque agent a sa propre logique de découverte de contexte — certains lisent le fichier automatiquement, d'autres nécessitent une configuration explicite. Répliquer l'étude sur plusieurs familles d'agents et de modèles fait partie de la feuille de route annoncée.

Les tâches testées sont volontairement petites : moins de 100 lignes, 5 fichiers maximum. Ce cadrage assure un protocole rigoureux et reproductible, mais laisse ouverte la question de l'effet sur des refactorisations majeures, des migrations de framework ou des développements multi-modules. L'intuition dit que le gain devrait être encore plus marqué sur des tâches complexes, où l'exploration du dépôt occupe une part plus grande du temps total — mais c'est une hypothèse, pas une mesure.

L'étude quantifie l'efficacité (temps et tokens), pas la qualité du code produit. La vérification manuelle sur 50 cas confirme l'absence de résultats dégénérés, mais une évaluation systématique de la correction sémantique, de la maintenabilité et de l'alignement avec l'intention du développeur reste à construire. Les chercheurs envisagent de comparer les changements générés par l'agent avec les PRs originales via des analyses de similarité structurelle et des suites de tests automatisées.

La piste de recherche la plus riche concerne le contenu du fichier lui-même. L'étude traite le fichier AGENTS.md comme une variable binaire : présent ou absent. Mais tous les fichiers ne se valent pas. Un AGENTS.md qui décrit l'architecture complète, avec des exemples de code et des anti-patterns à éviter, aura-t-il un effet différent d'un fichier qui liste simplement « npm run test » ? La granularité du contenu — spécificité, organisation, présence de conventions de codage versus simples commandes de build — pourrait moduler l'effet de façon considérable.

Les chercheurs avancent une hypothèse sur le mécanisme en jeu : un fichier AGENTS.md décrit la structure du dépôt et les conventions dès le départ, ce qui réduit le besoin de naviguer à l'aveugle dans l'arborescence. Moins d'exploration signifie moins de tours de raisonnement, moins de requêtes au modèle sous-jacent, et donc moins de tokens et de temps. Valider cette hypothèse nécessitera l'analyse fine des traces d'exécution — combien de fichiers l'agent ouvre-t-il avant de commencer à coder, combien de tentatives avortées précèdent la solution retenue.

Cette étude transforme une intuition de praticiens en donnée mesurable. Écrire un bon fichier d'instructions pour ses agents n'est pas un geste symbolique : c'est un levier d'optimisation quantifiable, à un coût dérisoire. Le travail reste préliminaire par sa taille d'échantillon, mais il pose les bases d'un champ de recherche neuf. Les fichiers AGENTS.md n'existaient pas il y a deux ans ; ils équipent déjà des dizaines de milliers de projets. Comprendre comment les rédiger, les maintenir et les évaluer devient un enjeu à part entière du génie logiciel assisté par IA.
