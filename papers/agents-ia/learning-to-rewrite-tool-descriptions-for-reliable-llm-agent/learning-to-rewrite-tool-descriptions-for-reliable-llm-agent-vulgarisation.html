Learning to Rewrite Tool Descriptions for Reliable LLM-Agent Tool Use

Vulgarisation

Quand les agents IA échouent, ce n'est pas toujours leur faute

Un agent IA reçoit une requête : « Trouve les films de Tom Hanks sortis après 2020 et récupère les critiques du mieux noté. » Pour y répondre, il doit enchaîner plusieurs appels à des APIs — une pour chercher les films, une pour les détails, une troisième pour les critiques. Mais comment sait-il quelle API appeler et avec quels paramètres ? Il lit leur description textuelle. Et c'est souvent là que tout déraille.

Une équipe d'Intuit AI Research (Guo, Dong, Gao et Das) vient de publier une approche qui améliore la performance des agents en réécrivant les descriptions d'outils — sans modifier l'agent lui-même. Leur framework, baptisé Trace-Free+, entraîne un modèle à produire de meilleures « notices » pour des APIs qu'il n'a jamais rencontrées, sans disposer de logs d'utilisation préalables. Sur certains benchmarks, les gains atteignent 25 points de pourcentage en taux de réussite.

Le problème : des notices écrites pour des humains, lues par des machines

Quand un développeur consulte la documentation d'une API, il mobilise son expérience, fait des inférences sur les cas limites, et teste en local si quelque chose est flou. Un agent LLM n'a pas ce luxe. Il prend la description au pied de la lettre.

Or les descriptions d'APIs sont écrites par et pour des humains. Elles contiennent des ambiguïtés, des informations manquantes sur les formats acceptés, des paramètres mal documentés. Quand l'agent doit choisir parmi cinq outils, ça passe encore. Mais les plateformes comme RapidAPI exposent des milliers d'endpoints. À cette échelle, une description vague ou trompeuse conduit l'agent à sélectionner le mauvais outil ou à générer des paramètres invalides.

Des travaux précédents ont montré qu'on pouvait améliorer ces descriptions en les révisant avec un LLM puissant. Certaines méthodes — DRAFT, Play2Prompt — vont plus loin : elles font tourner l'agent sur des requêtes de test, collectent les traces d'exécution (les logs de succès et d'échecs), puis utilisent ces retours pour affiner les descriptions.

Ces approches se heurtent à trois obstacles. D'abord, elles nécessitent des traces d'exécution, ce qui suppose de pouvoir appeler les APIs. Pour des outils fraîchement déployés (cold start), dans des environnements soumis à des contraintes de confidentialité, ou pour des APIs coûteuses, c'est impraticable. Ensuite, chaque outil est traité isolément : le système ne capitalise pas sur ce qu'il a appris en améliorant d'autres descriptions. Enfin, ces méthodes à base de prompts sont lentes, coûteuses en tokens, et difficiles à reproduire sur de nouveaux jeux de données.

La méthode : apprendre à rédiger de bonnes notices, puis se passer des retours terrain

L'idée centrale de Trace-Free+ repose sur une analogie directe. Imaginez un rédacteur technique junior à qui l'on confie des notices mal écrites accompagnées de rapports de bugs : « l'utilisateur a cru que le paramètre acceptait des dates, mais il faut un timestamp Unix ». Il apprend à repérer les pièges et à les anticiper dans sa rédaction. Puis, progressivement, on retire les rapports de bugs. Le rédacteur a internalisé les schémas d'erreurs fréquents et peut désormais les anticiper en lisant la spécification seule.

C'est le principe du curriculum learning appliqué ici. L'entraînement se déroule en deux phases. Au départ, le modèle reçoit la description originale de l'outil et un résumé des traces d'exécution — quels appels ont réussi, lesquels ont échoué, et pourquoi. Il doit produire une description améliorée. En seconde phase, la proportion d'exemples avec traces chute à 10 %, tandis que les exemples sans traces montent à 90 %. Le modèle apprend à générer les mêmes descriptions de qualité en se passant des traces au moment de l'inférence.

Mais pour entraîner ce modèle, il faut des données d'entraînement de qualité. Les chercheurs ont construit un pipeline en quatre étapes.

Le tri des outils. Parmi les 9 640 fournisseurs d'API de ToolBench — une collection d'APIs réelles issues de RapidAPI —, un agent automatisé explore chaque endpoint, teste des appels concrets, et étiquette leur état de santé. Résultat : seuls 1 234 fournisseurs ont toutes leurs APIs fonctionnelles. Après exclusion de ceux utilisés dans les benchmarks de test et de ceux comptant moins de trois endpoints, 107 fournisseurs restent. C'est le socle d'entraînement — à peine 1 % du catalogue initial, mais des APIs dont on sait qu'elles répondent correctement.

La synthèse de requêtes. Les chercheurs génèrent des questions utilisateur qui exigent d'enchaîner plusieurs APIs dans un ordre précis, avec des dépendances entre les résultats intermédiaires. Pas de requêtes triviales qu'un seul appel résout. Le LLM analyse d'abord les relations entre APIs d'un même fournisseur, sélectionne celles qui forment un workflow cohérent, puis formule une question en langage naturel qui nécessite implicitement toutes ces APIs.

La collecte de traces. Un agent exécute ces requêtes synthétiques et enregistre les trajectoires complètes — succès comme échecs. Les échecs révèlent les décalages entre ce que dit la description et ce que l'API attend réellement : un format de date incorrect, un paramètre obligatoire non documenté, une contrainte sur les valeurs acceptées.

L'amélioration en deux passes. Une première passe « générique » applique des bonnes pratiques de rédaction de documentation : clarté de l'intention, distinction paramètres obligatoires/optionnels, formats attendus. Puis une seconde passe « informée par les traces » intègre des règles extraites des échecs observés — par exemple : « le paramètre ip_address n'accepte que les formats IPv4 et IPv6 ». Ces descriptions finales servent de supervision pour le fine-tuning.

Le modèle retenu est un Qwen3-4B (quatre milliards de paramètres) fine-tuné avec LoRA sur huit GPUs A100. Un modèle compact, donc déployable sans infrastructure massive.

Résultats : des gains nets, y compris sur des domaines jamais vus

L'évaluation porte sur deux benchmarks : StableToolBench (APIs diversifiées couvrant 49 catégories) et RestBench (APIs TMDB pour le cinéma et Spotify pour la musique). Le modèle n'a vu aucun des outils de test pendant l'entraînement.

Le protocole d'évaluation utilise du teacher forcing : à chaque étape, le contexte est corrigé pour isoler l'effet des descriptions. Si l'agent échoue, c'est la description qui l'a induit en erreur, pas une erreur en cascade d'une étape précédente. C'est une mesure plus propre que les métriques habituelles.

Sur StableToolBench, en mode sans traces, Trace-Free+ atteint 54 % de taux de réussite au niveau des requêtes complètes, contre 48 % avec les descriptions originales. L'écart se creuse sur les requêtes multi-étapes les plus complexes (sous-ensemble G3) : 46,4 % contre 24,6 %. Le taux de réussite double presque.

La généralisation cross-domain livre le résultat le plus frappant. Le modèle, entraîné uniquement sur des APIs ToolBench, est testé sur TMDB et Spotify — des domaines totalement absents de l'entraînement. Sur TMDB, le taux de réussite par requête passe de 49,5 % (descriptions originales) à 74,9 %. Un bond de 25 points, sans aucune donnée spécifique au domaine cinéma. Sur Spotify, le gain est de 14 points (34,9 % → 49,3 %).

L'expérience de mise à l'échelle est tout aussi révélatrice. Quand on augmente le nombre d'outils candidats au-delà de 100, la plupart des méthodes voient leurs performances chuter brutalement. Trace-Free+ résiste mieux que toutes les alternatives : la baisse est la plus faible, et le modèle maintient un taux de réussite significatif même face à un large catalogue.

Au niveau des outils individuels, 6,8 % des outils voient leur score F1 de sélection s'améliorer grâce à Trace-Free+, contre 3,4 % qui se dégradent. Le modèle aide deux fois plus d'outils qu'il n'en pénalise.

En mode avec traces — quand on autorise l'exécution des outils avant déploiement — la version Trace-Based atteint 54,6 % au niveau requête sur StableToolBench, dépassant DRAFT (50 %) et Play2Prompt (52,5 %). Ces deux derniers requièrent des pipelines multi-tours élaborés pour collecter leurs traces ; le modèle entraîné ici n'a besoin que de traces simples collectées avec les descriptions originales.

Ce que ça change en pratique

Ce travail déplace le regard. La communauté s'est longtemps focalisée sur l'amélioration des modèles eux-mêmes — fine-tuning, prompting, chaînes de raisonnement. L'interface entre l'agent et ses outils restait un angle mort. Cette recherche montre qu'optimiser les descriptions d'outils est un levier complémentaire, applicable sans retrainer l'agent.

Pour les plateformes d'APIs, l'application est directe : un générateur de descriptions pourrait réécrire automatiquement la documentation des nouveaux endpoints au moment de leur publication, avant même qu'un seul utilisateur — humain ou IA — ne les appelle. Le cold start disparaît du problème.

Le choix d'un modèle de quatre milliards de paramètres n'est pas anodin. Un modèle de cette taille tourne sur un seul GPU grand public. Les fournisseurs d'APIs pourraient intégrer ce traitement dans leur pipeline CI/CD : à chaque mise à jour d'une spécification OpenAPI, le modèle régénère les descriptions optimisées pour les agents.

L'approche clarifie aussi la répartition des responsabilités. Quand un agent échoue à utiliser une API, le réflexe est de blâmer le modèle. Cette recherche montre qu'une part significative des échecs provient de l'interface elle-même. En améliorant les descriptions, on réduit la charge imposée au modèle et on lui libère de la capacité de raisonnement pour le problème réel de l'utilisateur.

Le protocole d'évaluation par teacher forcing représente aussi une contribution méthodologique. Les benchmarks existants pour les agents à outils mélangent les erreurs de description avec les erreurs en cascade dues à un contexte corrompu. Isoler l'effet de la description devrait devenir un standard dans l'évaluation des interfaces d'outils.

Pour aller plus loin

Plusieurs questions restent ouvertes. Le modèle actuel ne modifie que les descriptions textuelles — il ne corrige pas les schémas de paramètres en mode sans traces, car la tâche est jugée trop difficile sans retour d'exécution. Entraîner un modèle qui corrige simultanément descriptions et schémas est une piste naturelle.

L'optimisation fonctionne ici indépendamment de l'agent. Les descriptions améliorées bénéficient à n'importe quel LLM. Mais une co-optimisation — adapter les descriptions pour un agent spécifique tout en fine-tunant cet agent — pourrait amplifier les gains au-delà de ce que chaque approche obtient séparément.

Les expériences portent exclusivement sur des APIs REST. D'autres types d'outils — bases de données, environnements d'exécution de code, fonctions internes d'entreprise — présentent des caractéristiques différentes. La transférabilité de l'approche à ces contextes reste à vérifier.

Le curriculum learning à deux phases avec un ratio 10/90 (traces/sans traces) s'est révélé optimal sur ces benchmarks. Les auteurs ont testé trois phases sans amélioration. Comprendre pourquoi cette configuration spécifique fonctionne — et si elle reste optimale pour d'autres distributions d'outils — appelle des travaux théoriques complémentaires.

La question de fond que pose cette recherche dépasse le cadre technique : si la performance d'un système dépend autant de la qualité de ses interfaces que de l'intelligence de ses composants, l'effort d'optimisation devrait se répartir en conséquence. Les agents IA n'échappent pas à cette règle.
