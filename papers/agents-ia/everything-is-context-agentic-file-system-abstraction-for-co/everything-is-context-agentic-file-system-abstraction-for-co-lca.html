Everything is Context: Agentic File System Abstraction for Context Engineering

Lecture Critique d'Article

Lecture Critique d'Article

Publication analysée

Everything is Context: Agentic File System Abstraction for Context Engineering — Xu, Mao, Bai, Gu, Li, Zhu (CSIRO Data61, ArcBlock Inc., University of Tasmania) — arXiv, décembre 2025.

1. Objectif et contexte

L'ingénierie de contexte — comment capturer, structurer et gouverner les connaissances externes, la mémoire, les outils et les entrées humaines pour le raisonnement des LLM — manque d'un socle architectural unifié. Les pratiques actuelles (prompt engineering, RAG, intégration d'outils) produisent des artefacts transitoires, opaques et difficilement traçables. Le papier propose de transposer la philosophie Unix « everything is a file » à la gestion du contexte des agents IA : chaque ressource contextuelle (mémoire, outil, connaissance, contribution humaine) est montée dans un système de fichiers virtuel hiérarchique avec métadonnées et contrôle d'accès uniformes.

L'idée centrale est séduisante par sa simplicité : un espace de noms unique, des opérations standardisées (list, read, write, search), et une traçabilité par journalisation des transactions. Le problème adressé est réel — la fragmentation des pratiques de gestion de contexte dans les systèmes agentiques — mais la formulation reste celle d'un patron d'architecture plutôt que d'une question de recherche falsifiable.

2. Méthodologie

Le papier adopte une approche purement conceptuelle et architecturale. Aucun protocole expérimental n'est défini. La contribution se décline en trois volets :

  • Une abstraction de système de fichiers (AFS) appliquant les principes du génie logiciel (modularité, encapsulation, séparation des préoccupations, composabilité) à la gestion de contexte.
  • Un pipeline d'ingénierie de contexte en trois composants : Context Constructor (sélection et compression), Context Updater (injection et rafraîchissement dans la fenêtre de tokens), Context Evaluator (vérification des sorties et mise à jour du dépôt persistant).
  • Deux « exemplaires » d'implémentation dans le framework open-source AIGNE : un agent avec mémoire persistante et un assistant GitHub via MCP.

Les exemplaires se limitent à des extraits de code (listings de 16 et 20 lignes respectivement). Aucune mesure de performance, de qualité du contexte, de latence, de consommation de tokens, ou de comparaison avec des systèmes existants (LangChain, AutoGen, MemGPT, Zep) n'est présentée. Le papier ne définit aucune métrique d'évaluation. Les contraintes de conception identifiées (fenêtre de tokens, absence d'état, non-déterminisme) sont correctement formulées mais relèvent de connaissances établies plutôt que de contributions originales.

La taxonomie des types de mémoire (Table I) recense sept catégories (scratchpad, épisodique, factuelle, expérientielle, procédurale, utilisateur, historique) sans justification empirique de cette décomposition ni comparaison avec les taxonomies existantes dans la littérature cognitive ou en IA.

3. Résultats principaux

En l'absence d'évaluation expérimentale, les « résultats » se réduisent à la description de l'architecture et à la démonstration de faisabilité via le framework AIGNE :

  • L'AFS unifie les ressources hétérogènes sous un espace de noms cohérent avec des résolveurs programmatiques (similaires à GraphQL/OpenAPI).
  • Le module SystemFS supporte les opérations de base avec intégration ripgrep, métadonnées configurables et accès sandboxé.
  • Le Context Constructor génère un « manifeste de contexte » documentant les artefacts sélectionnés, exclus, et les raisons associées.
  • Le Context Evaluator déclenche une revue humaine lorsque les seuils de confiance sont bas ou que des contradictions sont détectées.

Ces éléments décrivent une architecture logicielle, pas des résultats de recherche. La question de savoir si cette abstraction améliore effectivement la traçabilité, la qualité du raisonnement, ou l'efficacité de la gestion de contexte par rapport aux approches existantes reste entièrement ouverte.

4. Discussion et limites

Le papier présente plusieurs faiblesses majeures :

L'absence totale d'évaluation empirique est le défaut le plus critique. Aucun benchmark, aucune étude utilisateur, aucune comparaison quantitative ne vient étayer les bénéfices revendiqués. Les affirmations sur la traçabilité, la gouvernance et la vérifiabilité restent programmatiques.

L'analogie avec le système de fichiers Unix, bien que pédagogiquement utile, n'est pas examinée dans ses limites. Un système de fichiers classique gère des données relativement statiques avec des opérations CRUD simples. Le contexte d'un agent LLM est dynamique, sémantiquement riche, et soumis à des contraintes de pertinence temporelle que l'abstraction fichier ne capture pas naturellement. Le papier ne discute pas de cette tension.

La distinction entre cette proposition et les travaux antérieurs (AIOS, MemGPT, Letta) reste floue. Le papier cite ces systèmes mais n'explique pas précisément ce que l'AFS apporte de différent au-delà d'une couche de nommage uniforme. MemGPT propose déjà une hiérarchie mémoire avec pagination ; AIOS implémente déjà des primitives OS pour les agents.

Le conflit d'intérêt potentiel mérite mention : quatre des six auteurs sont affiliés à ArcBlock Inc., l'entreprise derrière le framework AIGNE. Le papier fonctionne en partie comme une documentation architecturale de ce framework, ce qui pose question sur l'objectivité de l'évaluation (ou plutôt son absence).

La section VII (Conclusion and Future Work) ne contient aucune discussion des limitations. Les pistes futures (navigation agentique dans l'AFS, co-travail humain-IA renforcé) sont vagues et sans feuille de route concrète.

5. Reproductibilité

Le framework AIGNE est disponible en open-source sur GitHub (github.com/AIGNE-io/aigne-framework). Les extraits de code fournis sont suffisamment explicites pour reproduire les deux exemplaires. La stack technique (Node.js, SQLite, Docker pour le MCP GitHub) est standard.

Cependant, la reproductibilité au sens scientifique est impossible puisqu'il n'y a aucune expérience à reproduire, aucun jeu de données, aucun protocole de mesure. On peut instancier le framework, mais on ne peut pas vérifier les prétentions du papier sur la supériorité de l'approche.

6. Impact et applications

La contribution réside dans la formalisation d'un patron d'architecture pour l'ingénierie de contexte. L'idée de monter des ressources hétérogènes dans un espace de noms unifié avec métadonnées et contrôle d'accès est un patron de conception utile pour les développeurs de systèmes agentiques. Le pipeline Constructor/Updater/Evaluator offre un cadre mental structurant.

L'impact scientifique est limité par l'absence de validation. Pour les praticiens, le framework AIGNE peut avoir une utilité concrète, mais le papier n'apporte pas de preuve que cette abstraction surpasse les alternatives existantes en termes de qualité de raisonnement, de maintenabilité, ou de coût opérationnel.

L'intégration du protocole MCP (Model Context Protocol) comme module montable est un aspect pratique pertinent dans le contexte actuel de standardisation des interfaces outil-agent.

7. Positionnement dans la littérature

Le papier se situe à l'intersection du paradigme LLM-as-OS (AIOS, Mei et al. 2025 ; MemGPT, Packer et al. 2024) et de l'ingénierie de contexte émergente (LangChain 2024 ; Hua et al. 2025). La référence fondatrice au système Unix de Ritchie et Thompson (1974) est pertinente mais l'exécution ne va pas au-delà de l'analogie.

Comparé à MemGPT qui implémente une hiérarchie mémoire concrète avec pagination et évaluation, ou à Zep/Graphiti qui proposent des graphes de connaissance temporels avec benchmarks, ce papier reste au stade de la proposition architecturale. Les travaux de Dai et al. (2025) sur le framework intégré pour le context engineering et le ranking en cascade sont plus avancés en termes de validation empirique.

La littérature sur les systèmes de fichiers virtuels et sémantiques (Plan 9, FUSE, le semantic file system de Shi et al. 2025 cité dans le papier) offrait des bases de comparaison technique que les auteurs n'exploitent pas suffisamment.
