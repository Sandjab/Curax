Everything is Context: Agentic File System Abstraction for Context Engineering

Lecture Critique d'Article

Lecture Critique d'Article

Publication analysée

Everything is Context: Agentic File System Abstraction for Context Engineering
Xu, X., Gu, X., Mao, R., Li, Y., Bai, Q., Zhu, L. — arXiv:2512.05470 (cs.SE), décembre 2025

1. Objectif et contexte

L'ingénierie de contexte — la gestion structurée des connaissances, mémoires, outils et entrées humaines alimentant le raisonnement des LLM — manque de fondations architecturales unifiées. Les frameworks existants (LangChain, AutoGen, MemGPT) traitent mémoire, RAG et intégration d'outils comme des modules indépendants, produisant des artefacts transitoires difficiles à tracer et auditer.

Xu et al. proposent de transposer le principe Unix « everything is a file » au domaine du context engineering. Leur abstraction de système de fichiers (AFS) fournit un namespace hiérarchique où toutes les sources de contexte — bases vectorielles, graphes de connaissances, serveurs MCP, profils utilisateur — sont montées et accédées via une interface uniforme de lecture/écriture/recherche. Un pipeline de context engineering à trois composants (Constructor, Updater, Evaluator) orchestre la sélection, compression et validation du contexte sous contrainte de fenêtre de tokens.

L'implémentation repose sur AIGNE, un framework open-source développé par ArcBlock, Inc. Deux exemplars illustrent l'approche : un agent avec mémoire persistante et un assistant GitHub via MCP.

2. Méthodologie

Ce travail relève du position paper architectural. Aucun protocole expérimental n'est défini. La méthodologie se limite à :

  • Une analogie conceptuelle entre systèmes de fichiers Unix et gestion du contexte GenAI
  • Le découpage de l'architecture en principes de génie logiciel (abstraction, modularité, encapsulation, séparation des préoccupations, traçabilité, composabilité)
  • La description d'un pipeline à trois étages (Constructor → Updater → Evaluator)
  • Deux listings de code TypeScript (~20 lignes chacun) montrant l'API du framework AIGNE

L'absence de protocole expérimental est le défaut central du papier. Aucune hypothèse testable n'est formulée. Les auteurs ne définissent aucune métrique pour évaluer si l'abstraction file-system améliore la traçabilité, réduit la dérive contextuelle, ou facilite le développement par rapport aux approches existantes. Les deux exemplars sont des snippets d'initialisation d'agents — ils montrent une API, pas une évaluation.

La taxonomie des types de mémoire (Table I : scratchpad, épisodique, factuelle, expérientielle, procédurale, utilisateur, historique) est descriptive. Elle synthétise la littérature existante sans apporter de validation empirique sur l'utilité de cette catégorisation dans un système réel.

3. Résultats principaux

En l'absence d'expériences, les « résultats » sont des contributions architecturales :

  • Abstraction AFS : chaque source de contexte est projetée dans un namespace hiérarchique (/context/memory/, /context/tool/, /context/history/) via des résolveurs programmables. Les agents interagissent avec le contexte par des opérations list/read/write/search, indépendamment du backend de stockage.
  • Persistent Context Repository : cycle de vie History → Memory → Scratchpad avec transitions versionnées. L'historique est immuable et global ; la mémoire est spécifique à l'agent et mutable ; le scratchpad est éphémère et lié à une tâche.
  • Pipeline de context engineering : le Constructor sélectionne et compresse le contexte pertinent ; l'Updater synchronise la fenêtre de tokens en mode statique, incrémental ou adaptatif ; l'Evaluator vérifie les sorties et réintègre les résultats validés dans le repository.
  • Actions méta-définies : chaque fichier ou répertoire peut exposer des comportements appelables (résumé, validation, synchronisation), transformant les nœuds passifs en nœuds actifs.

Les deux exemplars de code montrent comment monter une mémoire SQLite et un serveur MCP GitHub dans l'espace AFS. Le code est lisible mais trivial — il démontre une API sans illustrer les mécanismes de sélection, compression ou évaluation du contexte.

4. Discussion et limites

Le papier souffre de faiblesses structurelles qui limitent sa portée scientifique.

Aucune validation. Pas de benchmark, pas d'étude utilisateur, pas de comparaison quantitative avec LangChain, AutoGen, MemGPT ou Letta. Les auteurs affirment que leur abstraction améliore la traçabilité et la gouvernance sans jamais le mesurer. La phrase « the implementation demonstrates how the architecture can be operationalised » résume le problème : démontrer qu'un code s'exécute n'est pas démontrer qu'il résout un problème mieux que les alternatives.

Exemplars insuffisants. Les deux listings de code sont des appels d'API d'initialisation. Ils ne montrent ni le fonctionnement du Constructor sous contrainte de tokens, ni le comportement de l'Updater en session multi-tours, ni la détection d'hallucinations par l'Evaluator. Les mécanismes centraux de l'architecture restent à l'état de description textuelle.

Métaphore non questionnée. L'analogie « everything is a file » est présentée comme naturellement bénéfique. Les auteurs ne discutent pas ses limites : les systèmes de fichiers Unix ont été critiqués pour leur sémantique trop pauvre (pas de typage, pas de relations), et des décennies de recherche en bases de données ont montré que tout réduire à des fichiers n'est pas toujours optimal. Cette réflexion est absente.

Conflit d'intérêts non déclaré. Trois des six auteurs travaillent chez ArcBlock, Inc., l'entreprise qui développe AIGNE. Le papier promeut directement ce framework sans le signaler comme une limitation potentielle de l'objectivité de l'évaluation (qui, de toute façon, n'existe pas).

Section limitations quasi inexistante. La conclusion mentionne des « future extensions » (navigation agentique, co-travail humain-IA renforcé) mais ne reconnaît aucune faiblesse de l'approche actuelle.

5. Reproductibilité

Le framework AIGNE est publié en open-source sur GitHub (github.com/AIGNE-io/aigne-framework). Les modules mentionnés (AFS, SystemFS, FSMemory, UserProfileMemory, MCPAgent) sont référencés avec leurs API. Les deux exemplars sont reproductibles en tant que code exécutable.

La reproductibilité scientifique est cependant sans objet : en l'absence d'expériences, il n'y a rien à reproduire au-delà de la compilation du code. Aucun jeu de données, aucun scénario de test, aucune métrique ne sont fournis pour qu'un tiers puisse évaluer indépendamment les bénéfices revendiqués.

6. Impact et applications

L'idée d'unifier la gestion du contexte GenAI via une abstraction de système de fichiers a un intérêt conceptuel réel. La séparation entre repository persistant et fenêtre de tokens bornée reflète une tension architecturale que tous les développeurs d'agents rencontrent. Le pipeline Constructor/Updater/Evaluator formalise des étapes que les praticiens effectuent de manière ad hoc.

L'impact reste cependant hypothétique. Sans données montrant que cette abstraction réduit les erreurs de contexte, améliore la cohérence multi-tours, ou simplifie le développement par rapport à LangChain ou AutoGen, l'adoption repose uniquement sur l'attrait de la métaphore. Le cadre « human-in-the-loop » est mentionné à plusieurs reprises mais jamais instancié concrètement — les mécanismes de vérification humaine et d'injection de connaissances tacites restent vagues.

7. Positionnement dans la littérature

Le papier se situe à l'intersection du paradigme LLM-as-OS (AIOS, Ge et al. 2023 ; Mei et al. 2025), des architectures de mémoire pour agents (MemGPT/Letta, Packer et al. 2024 ; Zep/Graphiti, Rasmussen et al. 2025), et de l'ingénierie de contexte émergente (Mei et al. 2025 ; Hua et al. 2025).

Par rapport à AIOS, qui propose des primitives OS (scheduling, allocation de ressources) pour systèmes multi-agents, ce travail se concentre sur la couche de gestion du contexte plutôt que sur l'orchestration de processus. Par rapport à MemGPT, qui implémente une hiérarchie mémoire court-terme/long-terme, l'abstraction AFS est plus générale (elle englobe aussi les outils et les sources externes) mais moins concrète (MemGPT a des benchmarks sur la rétention conversationnelle).

La bibliographie est récente (2023-2025) et couvre les travaux pertinents, mais reste dominée par des références à des blogs et documentations de frameworks (LangChain, Anthropic, Chroma) plutôt qu'à des publications évaluées par les pairs. Sur 30 références, environ un tiers sont des sources non académiques.
