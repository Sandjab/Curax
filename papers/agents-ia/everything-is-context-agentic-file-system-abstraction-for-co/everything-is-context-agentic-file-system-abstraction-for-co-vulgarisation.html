Everything is Context: Agentic File System Abstraction for Context Engineering

Vulgarisation

Introduction

ChatGPT, Claude, Gemini : ces modèles de langage traitent des millions de requêtes par jour. Mais chaque fois qu'un utilisateur lance une conversation, le modèle repart de zéro. Aucune mémoire de la session précédente, aucun accès automatique aux documents pertinents, aucune trace de ce qui a déjà été dit ou décidé. Toute l'intelligence apparente repose sur ce qu'on glisse dans une fenêtre de contexte — un espace limité, souvent à 128 000 ou 200 000 tokens. C'est comme demander à un expert de résoudre un problème complexe en ne lui autorisant qu'une pile de Post-it sur son bureau : il faut choisir quels Post-it garder, lesquels jeter, et dans quel ordre les empiler.

Une équipe de chercheurs du CSIRO (Australie), de l'Université de Tasmanie et de la société ArcBlock propose une approche architecturale pour structurer cette gestion du contexte. Leur idée : s'inspirer d'un principe vieux de cinquante ans en informatique — la philosophie Unix « tout est un fichier » — et l'appliquer aux systèmes d'IA générative. Le résultat est une abstraction de système de fichiers pour l'ingénierie du contexte, implémentée dans un framework open source nommé AIGNE.

Le problème : un contexte fragile, dispersé et opaque

Les systèmes d'IA générative actuels accumulent les rustines pour gérer le contexte. Le prompt engineering permet de formuler des instructions précises. Le RAG (Retrieval-Augmented Generation) va chercher des documents dans une base vectorielle. Les intégrations d'outils connectent le modèle à des API externes. Chaque technique fonctionne, mais de façon isolée. Aucune ne partage un format commun avec les autres. Aucune ne conserve de trace vérifiable de ce qui a été injecté dans le modèle, à quel moment, et pourquoi.

Cette fragmentation crée deux problèmes concrets. Le premier : le context rot, la dégradation progressive du contexte au fil du temps. Les informations deviennent obsolètes, les résumés perdent en fidélité, les mémoires se désynchronisent. Le second : l'absence de traçabilité. Quand un agent IA produit une réponse erronée, il est souvent impossible de remonter la chaîne pour comprendre quelles informations lui ont été fournies et lesquelles ont été omises.

Les frameworks existants — LangChain, AutoGen, mem0 — apportent des briques utiles pour la mémoire, la recherche ou l'orchestration d'outils. Mais ils traitent ces composants comme des modules indépendants, sans fondation architecturale commune. Chaque projet réinvente sa propre plomberie, avec des artefacts de contexte souvent éphémères et impossibles à auditer.

La méthode : quand l'IA emprunte au système de fichiers

La proposition des chercheurs tient en une analogie forte. Dans les systèmes Unix, tout — disques durs, imprimantes, connexions réseau — est accessible via une interface de fichier. On lit, on écrit, on parcourt, avec les mêmes commandes, quel que soit le périphérique réel derrière. Cette uniformité a permis de construire des systèmes modulaires et composables pendant cinquante ans.

Les auteurs appliquent ce principe à l'IA générative. Mémoire conversationnelle, bases de connaissances, outils externes, annotations humaines, profils utilisateurs : tout devient un « fichier » dans un système de fichiers virtuel. Un agent IA qui a besoin d'une information n'a pas à savoir si elle provient d'une base vectorielle, d'un graphe de connaissances ou d'un fichier SQLite. Il accède à un chemin — /context/memory/fact/ par exemple — et le système se charge du reste.

Ce système de fichiers agentique (AFS, pour Agentic File System) fournit quatre opérations de base : lister, lire, écrire, rechercher. Chaque ressource est « montée » dans l'arborescence, exactement comme on monte un disque dur externe sous Linux. Un serveur MCP (Model Context Protocol) pour GitHub ? Il apparaît sous /modules/github-mcp/. La mémoire d'un agent ? Sous /context/memory/agentID/. Les annotations humaines ? Sous /context/human/.

Cette uniformité élimine le besoin d'écrire du code d'intégration spécifique pour chaque source de données. Les schémas déclaratifs — similaires à GraphQL ou OpenAPI — projettent automatiquement les structures internes sur des nœuds du système de fichiers. Ajouter une nouvelle source revient à la monter, comme on brancherait une clé USB.

Le dépôt de contexte persistant

Les modèles de langage sont amnésiques par nature. Pour compenser, l'architecture définit un dépôt persistant organisé en trois couches. L'historique (History) enregistre toutes les interactions brutes de façon immuable — c'est la boîte noire du système. La mémoire (Memory) stocke des vues structurées et indexées : résumés d'épisodes, faits atomiques, préférences utilisateur, trajectoires d'expérience. Chaque type de mémoire a sa propre durée de vie et sa propre granularité. Enfin, le bloc-notes (Scratchpad) sert d'espace de travail temporaire pendant le raisonnement — les brouillons d'un agent, en quelque sorte.

La transition entre ces couches suit un cycle de vie précis. Les données brutes arrivent dans l'historique. Des processus de résumé et d'indexation les transforment en entrées mémoire. Pendant le raisonnement, l'agent griffonne sur son bloc-notes. À la fin d'une session, les éléments pertinents du bloc-notes migrent vers la mémoire ou sont archivés dans l'historique. Chaque transformation est horodatée et versionnée — on peut rejouer n'importe quelle étape.

Le pipeline d'ingénierie du contexte

Le système de fichiers fournit l'infrastructure. Par-dessus, les chercheurs construisent un pipeline en trois étapes qui orchestre le flux de contexte vers le modèle.

Le Context Constructor sélectionne les éléments pertinents dans le dépôt persistant. Il consulte les métadonnées — date de création, provenance, périmètre d'accès — pour prioriser les informations. Il compresse ensuite le contexte retenu (par résumé, par regroupement, par extraction d'embeddings) afin de respecter le budget de tokens. À chaque construction, il génère un manifeste : la liste de ce qui a été inclus, exclu, et les raisons de chaque décision. Ce manifeste transforme l'assemblage du contexte, habituellement une opération invisible, en un processus traçable.

Le Context Updater gère l'injection du contexte dans la fenêtre du modèle pendant le raisonnement. Pour une tâche ponctuelle, il injecte un instantané statique. Pour une conversation longue, il rafraîchit progressivement le contenu : les fragments obsolètes sont remplacés, de nouvelles informations sont ajoutées au fil du dialogue. Ce composant surveille en permanence la taille du contexte actif pour ne jamais dépasser la limite de tokens.

Le Context Evaluator ferme la boucle. Après chaque réponse du modèle, il évalue la sortie en la confrontant aux éléments de contexte qui l'ont produite. Il cherche les hallucinations, les contradictions, les dérives par rapport aux sources. Quand le niveau de confiance tombe sous un seuil, il déclenche une vérification humaine. Les corrections et annotations des humains sont alors stockées comme des éléments de contexte à part entière — le savoir tacite des experts devient une ressource de première classe dans le système.

Résultats clés

L'architecture est implémentée dans le framework open source AIGNE, compatible avec les principaux modèles du marché (OpenAI, Claude, Gemini, DeepSeek, Ollama). Deux démonstrateurs illustrent son fonctionnement.

Le premier démonstrateur est un agent doté de mémoire persistante. En quelques lignes de code, un développeur monte un historique de conversation et un profil utilisateur sur un fichier SQLite. L'agent conserve automatiquement le fil de la discussion d'une session à l'autre, sans gestion d'état manuelle. Ce qui prenait auparavant des dizaines de lignes de code ad hoc — sérialisation, désérialisation, gestion de la taille du contexte — se réduit à un montage déclaratif.

Le second démonstrateur connecte un serveur MCP GitHub au système de fichiers. Une fois monté, l'agent interagit avec les dépôts GitHub (recherche de repositories, liste d'issues) comme s'il parcourait des répertoires locaux. L'ajout de n'importe quel autre serveur MCP — Slack, Jira, une base de données interne — suit le même patron : un montage, et toutes les capacités deviennent accessibles via l'interface unifiée.

L'article ne présente pas de benchmarks comparatifs chiffrés avec les frameworks existants. La contribution est architecturale : elle propose un modèle de conception, pas une optimisation de performance. La validation repose sur la démonstration de faisabilité et sur l'alignement avec des principes éprouvés du génie logiciel — modularité, encapsulation, séparation des préoccupations, traçabilité.

Implications pratiques

Pour les équipes qui construisent des agents IA en production, la gestion du contexte est un cauchemar d'ingénierie. Chaque nouveau projet assemble sa propre combinaison de bases vectorielles, de caches mémoire, de connecteurs d'outils et de stratégies de compression. Quand un agent produit une réponse incorrecte, le débogage ressemble à une archéologie : quelles informations étaient dans le prompt ? Quelle version de la mémoire a été utilisée ? Un document a-t-il été tronqué ?

L'approche par système de fichiers apporte une réponse structurelle à ces questions. La traçabilité intégrée — chaque opération est journalisée avec son horodatage, sa source et sa lignée — permet de reconstituer exactement l'état du contexte à n'importe quel moment. Les équipes peuvent appliquer au contexte les mêmes pratiques DevOps qu'elles utilisent déjà pour le code : versionnement, revue, déploiement contrôlé.

La dimension humaine de l'architecture mérite attention. Les systèmes IA actuels relèguent souvent l'humain au rôle de superviseur passif qui valide ou rejette des sorties. Ici, les humains sont des participants actifs du pipeline de contexte. Leurs corrections, leurs annotations, leur expertise tacite sont stockées dans le système de fichiers au même titre que les données automatiques. Cette symétrie entre contributions humaines et machiniques pose une base concrète pour des systèmes où l'IA et l'humain collaborent réellement, plutôt que de fonctionner en séquence.

L'interopérabilité via le protocole MCP est un autre levier pratique. Le standard MCP, poussé par Anthropic et adopté par un nombre croissant d'outils, permet à n'importe quel service de s'exposer comme un module contextuel. L'architecture AFS transforme chaque serveur MCP en un répertoire navigable. Pour une entreprise qui utilise déjà des serveurs MCP pour ses outils internes, l'adoption de cette architecture ne nécessite pas de réécrire les connecteurs existants — elle les unifie sous une interface commune.

Pour aller plus loin

L'article ouvre plusieurs directions de recherche. La plus ambitieuse : permettre aux agents de naviguer eux-mêmes dans l'arborescence du système de fichiers, de construire leurs propres index, de réorganiser leurs structures de données. L'agent ne serait plus un consommateur passif de contexte préparé pour lui — il deviendrait un processus auto-organisé qui observe et modifie son propre environnement informationnel. Les auteurs parlent d'un « tissu de connaissances vivant » où raisonnement, mémoire et action convergent dans un substrat vérifiable.

Plusieurs questions restent sans réponse. La scalabilité de l'approche en environnement multi-agents massif n'est pas évaluée. Quand des centaines d'agents partagent le même système de fichiers, les problèmes de concurrence, de cohérence et de performance deviennent critiques. Les politiques de gouvernance — qui peut lire quoi, combien de temps un souvenir est conservé, comment résoudre les conflits entre mémoires contradictoires — sont esquissées mais pas formalisées.

L'absence de benchmarks quantitatifs limite aussi la portée des conclusions. L'architecture est élégante sur le papier, mais comment se compare-t-elle à LangChain ou AutoGen en termes de latence de récupération, de qualité des réponses, de coût en tokens ? Ces mesures seront nécessaires pour convaincre les équipes d'ingénierie de migrer vers ce modèle.

Enfin, la tension entre compression et fidélité reste un défi ouvert. Résumer un document pour respecter le budget de tokens, c'est accepter une perte d'information. Le pipeline propose des mécanismes de compression, mais la question de savoir quand cette perte devient critique — quand un résumé trop agressif conduit à une hallucination — n'a pas encore de réponse satisfaisante dans la littérature.

L'approche « tout est fichier » pour le contexte IA ne résout pas tous ces problèmes. Elle fournit une fondation — une grammaire commune — sur laquelle ces solutions pourront se construire. Et dans un domaine où chaque projet réinvente sa propre plomberie, disposer d'une fondation partagée est déjà un progrès substantiel.
