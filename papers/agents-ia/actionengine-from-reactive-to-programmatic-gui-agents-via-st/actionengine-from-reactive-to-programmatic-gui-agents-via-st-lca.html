ActionEngine : From Reactive to Programmatic GUI Agents via State Machine Memory

Zhong, H., Faisal, F., França, L., Leesatapornwongsa, T., Szekeres, A., Rong, K., Nath, S. — Georgia Tech & Microsoft Research, 2026

1. Objectif et contexte

Les agents GUI autonomes — systèmes capables de piloter des interfaces graphiques pour accomplir des tâches utilisateur — reposent aujourd'hui sur un cycle réactif : à chaque étape, un modèle de langage multimodal (MLLM) observe une capture d'écran, détermine l'action suivante et l'exécute. Ce cycle observer-raisonner-agir engendre un coût linéaire en O(N) appels au modèle pour N étapes, une latence proportionnelle, et une accumulation d'erreurs causée par les hallucinations visuelles. Sur le benchmark WebArena, les meilleurs agents réactifs plafonnent autour de 58-66 % de réussite.

ActionEngine propose un changement de paradigme : remplacer le raisonnement itératif par une planification programmatique en un seul appel LLM. L'idée repose sur une séparation nette entre l'apprentissage de la structure de l'application (hors ligne) et l'exécution des tâches (en ligne). Un agent de crawl construit un graphe de machine à états (State Machine Graph, SMG) modélisant la topologie de l'application cible. Un agent d'exécution exploite ce graphe pour synthétiser un programme Python complet résolvant la tâche demandée.

2. Méthodologie

L'architecture se décompose en trois phases distinctes.

Le Crawling Agent explore l'application cible hors ligne et construit un graphe dirigé. Les nœuds représentent des états symboliques (pages ou fenêtres), identifiés par leur signature structurelle — une combinaison d'atomes statiques (barre de navigation, filtres) et dynamiques (contenus variables). Les arêtes encodent des opérations exécutables : manipulations UI provoquant des transitions (cliquer sur un bouton) ou collectes de données sans changement d'état (lire les commentaires). Pour éviter l'explosion combinatoire, le graphe distingue les templates des instances : deux pages de forums différents partagent le même nœud si leur structure est identique. Le graphe Reddit résultant contient 20-30 états et 100-150 transitions.

L'Execution Agent reçoit une tâche en langage naturel et un sous-graphe filtré du SMG. La génération s'effectue en trois étapes : (1) un sketch program — programme Python intermédiaire avec des placeholders UI_CALL pour les interactions GUI ; (2) un linking statique qui résout chaque placeholder en un chemin concret dans le graphe par recherche en largeur (BFS) ; (3) une compilation en plan d'actions mixte typé (UINodes, PythonNodes, ControlFlowNodes) exécutable via Playwright.

Un mécanisme d'adaptation dynamique intervient en cas d'échec d'exécution. Un validateur distingue les erreurs Python (hot-patching de code) des erreurs UI (fallback vers un agent visuel qui recapture le DOM et met à jour le SMG). Les corrections réussies sont persistées dans le graphe pour les exécutions futures.

L'évaluation porte sur le sous-ensemble Reddit (Postmill) de WebArena : 106 tâches nécessitant navigation inter-pages, filtrage logique et arithmétique. Le seul baseline comparé est AgentOccam, un agent réactif état de l'art utilisant GPT-4-Turbo. ActionEngine utilise Claude 4.5 Sonnet.

3. Résultats principaux

ActionEngine atteint 95 % de réussite sur les 106 tâches Reddit, contre 66 % pour AgentOccam. La latence moyenne passe de 237 s à 118 s, le coût par tâche de 0,71 $ à 0,06 $, et le nombre moyen d'appels LLM de 10,2 à 1,8.

Les gains les plus marqués apparaissent sur les tâches de raisonnement multi-étapes. Le groupe 27-31 (comptage de commentaires filtrés par critères de votes) illustre la différence : AgentOccam consomme 140,9k tokens d'entrée en 17,8 appels LLM pour un taux de réussite de 60 %, tandis qu'ActionEngine résout ces tâches en un seul appel LLM avec 6,8k tokens et 100 % de réussite. Sur les tâches simples (profil utilisateur, groupe 399-403), ActionEngine réussit 100 % là où AgentOccam échoue totalement — un résultat surprenant qui suggère que même les tâches élémentaires posent problème au paradigme réactif lorsqu'elles requièrent une navigation précise.

Les 6 échecs d'ActionEngine (tâches 409, 599, 617, 625, 715, 722) relèvent tous d'ambiguïtés dans l'énoncé des tâches : correspondance floue de noms de forums ("machinelearning" vs "MachineLearning"), interprétation du mot "history" comme forum ou comme historique global, identification du "manager of the website" sans indice explicite dans les commentaires. Aucun échec n'est attribuable à une défaillance architecturale du système.

4. Discussion et limites

Le biais expérimental le plus sérieux concerne l'asymétrie des modèles. ActionEngine emploie Claude 4.5 Sonnet ; AgentOccam repose sur GPT-4-Turbo. Ces modèles diffèrent en capacités de génération de code, en compréhension visuelle et en tarification. Le gain de 11,8× en coût mélange l'effet de l'architecture avec l'effet du prix par token. Aucune ablation n'isole la contribution du SMG de celle du choix de modèle. Une expérience où les deux systèmes utilisent le même LLM aurait fourni une comparaison bien plus informative.

L'évaluation se cantonne à un seul domaine (Reddit/Postmill) d'un seul benchmark (WebArena). Les auteurs justifient ce choix par la complexité du raisonnement long horizon sur Reddit, mais cette restriction interdit toute conclusion sur la généralisabilité. Comment le SMG se comporte-t-il sur un site e-commerce avec des millions de pages produits ? Sur une application mobile avec des gestes tactiles ? Sur une SPA dont le DOM change sans rechargement de page ? Ces questions restent ouvertes.

Le coût du crawl hors ligne est absent des métriques comparatives. Le processus est qualifié de "semi-automated" : un MLLM identifie les atomes et opérations, mais une validation humaine intervient. Combien de temps prend la construction d'un SMG Reddit de 150 transitions ? Combien d'appels MLLM le crawl consomme-t-il ? L'amortissement sur N tâches est plausible, mais sans chiffres, le lecteur ne peut évaluer la rentabilité réelle.

Le mécanisme de fallback visuel, déclenché lors d'échecs d'exécution, réintroduit ponctuellement la dépendance aux MLLM que l'architecture cherche à éliminer. Les auteurs rapportent qu'avec Claude 4.5 Sonnet, le validateur intervient rarement, mais qu'avec GPT-4o, 20-30 % des exécutions nécessitent un hot-patching. La robustesse du système dépend donc encore de la qualité du LLM planificateur — un couplage que l'architecture n'abolit pas entièrement.

L'hypothèse de décomposabilité en templates stables est forte. Les applications web modernes — avec rendu côté client, chargement paresseux, A/B testing permanent — produisent des structures de page variables qui pourraient dégrader la stabilité du SMG.

5. Reproductibilité

Aucun dépôt de code n'est mentionné. Aucun SMG pré-construit n'est distribué. Les prompts du Crawling Agent et de l'Execution Agent ne sont pas intégralement reproduits — seuls des extraits YAML et des listings simplifiés figurent dans l'article. Le processus de construction du graphe requiert une intervention humaine dont l'ampleur n'est pas quantifiée. Le format YAML du SMG est illustré (Listing 1), mais un lecteur souhaitant reproduire l'expérience devrait reconstruire l'ensemble du pipeline à partir de descriptions textuelles. La reproductibilité est faible en l'état.

6. Impact et applications

Le passage d'un paradigme réactif O(N) à un paradigme programmatique O(1) représente un changement architectural concret pour l'automatisation GUI. La réduction de coût de ×11,8 rend envisageable un déploiement en production pour des workflows répétitifs sur des applications à structure stable — support client, tests de régression, scraping structuré. Le découplage crawl/planification/exécution permet de maintenir chaque composant indépendamment : mettre à jour le SMG sans modifier le planificateur, changer de LLM sans recrawler.

Le mécanisme d'auto-correction du graphe par feedback d'exécution est un apport pratique : le système apprend des échecs ponctuels sans nécessiter de re-crawl complet. Ce design s'apparente à un cache adaptatif d'interactions GUI.

La limitation au domaine Reddit laisse ouverte la question de la scalabilité. Un déploiement industriel devrait couvrir des dizaines d'applications simultanément, chacune nécessitant son propre SMG — un investissement initial non négligeable.

7. Positionnement dans la littérature

ActionEngine se distingue des agents GUI réactifs (SeeClick, ScreenAgent, AgentOccam) par l'introduction d'une mémoire structurée exploitée pour la planification globale plutôt que pour le contexte local. Synapse stocke des trajectoires passées comme exemples de prompts — une mémoire plate sans structure topologique. PageGraph modélise les transitions entre pages mais ignore les opérations d'extraction de données. AutoDroid utilise un graphe de transitions UI pour enrichir le contexte d'un agent qui reste réactif. ActionEngine franchit une étape supplémentaire : le graphe ne sert plus d'aide au raisonnement mais de substrat pour la synthèse de programmes déterministes.

L'approche de compilation (sketch → linking → plan exécutable) rappelle les techniques classiques de synthèse de programmes avec spécification partielle. L'analogie avec les vues matérialisées en bases de données, mentionnée par les auteurs pour les opérations composées fréquentes, est pertinente mais non implémentée dans le prototype actuel.

Par rapport aux travaux sur les modèles du monde pour agents web (WebDreamer), ActionEngine adopte une stratégie différente : plutôt que de simuler les conséquences d'actions possibles, le système encode la structure réelle de l'application dans un artefact réutilisable. Cette approche élimine l'incertitude de la simulation au prix d'un investissement initial de crawl.
