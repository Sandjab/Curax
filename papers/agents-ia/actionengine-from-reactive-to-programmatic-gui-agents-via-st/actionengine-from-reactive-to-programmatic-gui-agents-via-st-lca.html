ActionEngine: From Reactive to Programmatic GUI Agents via State Machine Memory

Lecture Critique d'Article

Objectif et contexte

Les agents GUI actuels fonctionnent selon un schéma réactif : à chaque étape, un modèle multimodal observe une capture d'écran, décide de l'action suivante, l'exécute, puis recommence. Ce cycle observe–raisonne–agit engendre un coût linéaire en appels LLM (O(N) pour N actions) et une accumulation d'erreurs visuelles le long de la trajectoire. ActionEngine propose de remplacer ce raisonnement itératif par une planification programmatique en une passe unique.

L'architecture repose sur deux agents spécialisés. Un Crawling Agent explore l'application cible hors-ligne et construit un graphe de machine à états (State Machine Graph, SMG) — un graphe dirigé où les nœuds représentent des états de page et les arêtes des opérations GUI concrètes. Un Execution Agent exploite ensuite ce graphe pour synthétiser un programme Python complet en un seul appel LLM, exécuté de façon déterministe sans inférence supplémentaire.

Le problème adressé est réel et bien documenté : les benchmarks WebArena, Mind2Web et AgentBench montrent que les agents réactifs les plus avancés plafonnent autour de 58 % de réussite. La séparation entre apprentissage de la structure applicative et planification de tâche répond à une limitation architecturale reconnue.

Méthodologie

Construction du graphe de machine à états

Le Crawling Agent parcourt l'application à partir d'un état initial et identifie, via un MLLM, les « atomes » — groupes d'éléments GUI qui apparaissent ou disparaissent ensemble. Chaque état est défini par sa signature d'atomes. Les atomes statiques (barre de navigation) se distinguent des atomes dynamiques (contenu variable), ce qui permet de regrouper des pages structurellement identiques mais peuplées de données différentes sous un même état template. Pour Reddit, le graphe résultant contient 20 à 30 états et 100 à 150 transitions.

Le processus est qualifié de « semi-automatisé » sans que l'article ne quantifie l'intervention humaine requise. Le coût en appels LLM de cette phase de crawling n'apparaît dans aucune mesure.

Planification et compilation

L'Execution Agent produit d'abord un programme-esquisse (sketch program) en Python contenant des placeholders UI_CALL. Un linker résout ces placeholders par recherche BFS dans le SMG. Un compilateur transforme le tout en un MixedActionPlan — arbre de nœuds typés (UI, Python, contrôle de flux) exécutables via Playwright. Un mécanisme de fallback basé sur la vision intervient en cas d'échec d'exécution et met à jour le graphe.

Protocole d'évaluation

L'évaluation porte sur le sous-ensemble Reddit (Postmill) de WebArena : 106 tâches sur les ~812 du benchmark complet. Le seul baseline est AgentOccam, un agent réactif utilisant GPT-4-Turbo. ActionEngine utilise Claude 4.5 Sonnet. Les métriques rapportées sont le taux de réussite, la latence, le coût par tâche, le nombre de tokens et le nombre d'appels LLM.

Résultats principaux

ActionEngine atteint 95 % de réussite sur les 106 tâches Reddit contre 66 % pour AgentOccam. La latence moyenne passe de 237 s à 118 s (÷2), le coût moyen de 0,71 $ à 0,06 $ (÷11,8), et le nombre d'appels LLM de 10,2 à 1,8 par tâche. Sur les tâches de raisonnement multi-étapes (groupe 27-31), le système passe de 17,8 appels LLM et 140,9k tokens d'entrée à 1 appel et 6,8k tokens.

Six tâches échouent. L'analyse détaillée révèle que cinq relèvent d'ambiguïtés dans la spécification des tâches (interprétation de « manager of the website », matching exact de noms de forums) et une d'une limitation technique (absence d'outil d'analyse d'image). Les traces de raisonnement du LLM sont reproduites intégralement pour chaque échec — un effort de transparence rare.

Discussion et limites

Comparaison inéquitable entre modèles

Le problème méthodologique central : ActionEngine utilise Claude 4.5 Sonnet tandis qu'AgentOccam utilise GPT-4-Turbo. Ces deux modèles diffèrent en capacités de raisonnement, en génération de code, et en tarification. Le gain de 11,8× en coût mélange trois effets non démêlés : l'architecture (moins d'appels), le choix du modèle (tarification différente), et la capacité du modèle (meilleur code en une passe). Sans exécuter AgentOccam avec Claude 4.5 Sonnet — ou ActionEngine avec GPT-4-Turbo — la part attribuable à l'architecture reste inconnue.

L'article mentionne d'ailleurs que le Validator n'intervient presque jamais avec Claude 4.5 Sonnet mais corrige 20-30 % des exécutions avec GPT-4o. Cela confirme que la performance dépend lourdement du modèle sous-jacent, pas seulement de l'architecture.

Évaluation sur un seul domaine

WebArena comprend cinq domaines : Reddit, Shopping (OneStopShop), CMS (GitLab), Maps et un forum. L'évaluation ne couvre que Reddit, soit environ 13 % du benchmark. Le sous-ensemble choisi est présenté comme le « pire cas pour le raisonnement à long horizon », mais cette affirmation n'est pas étayée par une comparaison avec les autres domaines. Les applications e-commerce ou cartographiques présentent des structures bien plus dynamiques et des espaces d'états considérablement plus grands — la généralisation de l'approche à ces contextes reste hypothétique.

Coût du crawling occulté

La phase de crawling mobilise un MLLM pour identifier les atomes, énumérer les opérations candidates et valider les transitions. Ce coût — potentiellement élevé pour une application complexe — n'est ni mesuré ni amorti dans les comparaisons. Le temps humain pour la partie « semi-automatisée » n'est pas non plus quantifié. L'argument d'efficacité porte uniquement sur la phase d'exécution, ce qui biaise la comparaison en faveur du système proposé.

Fragilité face aux évolutions majeures d'interface

Le mécanisme de mise à jour dynamique gère les changements localisés (un sélecteur CSS modifié). L'article reconnaît qu'une refonte majeure de l'interface nécessite un re-crawling complet, mais ne teste pas ce scénario. Pour des applications web mises à jour fréquemment, la maintenance du graphe pourrait annuler les gains d'efficacité.

Absence de tests statistiques

Aucun intervalle de confiance, aucune mesure de variance, aucun test de significativité. Les résultats sont rapportés comme des moyennes brutes. Sur 106 tâches, l'écart de 29 points de pourcentage entre les deux systèmes est probablement significatif, mais l'article ne le démontre pas formellement.

Reproductibilité

Aucun dépôt de code, aucun artefact de données, aucun graphe SMG partagé. Le processus de crawling semi-automatisé implique des décisions humaines non documentées. Les prompts utilisés pour le Planner et le Crawling Agent ne sont pas reproduits (seuls des extraits de YAML et de code sont montrés). La reproduction indépendante de ces résultats serait difficile sans accès au code source et aux graphes construits.

L'utilisation de WebArena comme benchmark est un point positif : l'environnement est public et standardisé. Mais le code spécifique à ActionEngine manque pour permettre une réplication.

Impact et applications

L'idée de transformer l'interaction GUI en traversée de graphe déterministe, avec synthèse de programme en une passe, a un potentiel applicatif réel pour l'automatisation d'entreprise (RPA). La réduction drastique des appels LLM rend le déploiement économiquement viable pour des tâches répétitives sur des applications stables.

Les limites pratiques sont cependant significatives. Le crawling semi-automatisé freine le passage à l'échelle vers de nouvelles applications. L'hypothèse d'une structure GUI relativement stable n'est pas toujours vérifiée (applications SPA modernes, A/B testing permanent). Le système reste démontré sur un seul type d'application — un forum — dont la structure hiérarchique (accueil → forums → posts → commentaires) est particulièrement adaptée à une modélisation par machine à états.

Positionnement dans la littérature

ActionEngine se distingue des agents réactifs (SeeClick, ScreenAgent, AgentOccam) par son passage à la planification programmatique. Par rapport aux systèmes à mémoire existants — Synapse stocke des trajectoires plates, PageGraph modélise les transitions de pages, AutoDroid annote les éléments UI — le SMG capture à la fois les opérations de navigation et les opérations de collecte de données. Cette distinction est pertinente : les self-loops pour les opérations de lecture permettent de modéliser l'extraction d'information sans changement d'état.

L'approche rappelle les travaux en synthèse de programme et en planification symbolique, mais appliqués au domaine GUI avec un LLM comme générateur de code. Le rapprochement avec les « vues matérialisées » pour les séquences d'opérations fréquentes est judicieux, bien que non implémenté ni évalué.

La comparaison avec un seul baseline affaiblit le positionnement. WebVoyager, WebDreamer, MindAct et d'autres agents récents auraient fourni des points de référence supplémentaires sur les mêmes tâches Reddit. L'absence de ces comparaisons laisse ouverte la question de la supériorité réelle de l'architecture par rapport aux progrès récents des agents réactifs multimodaux.
