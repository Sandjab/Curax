VibeTensor: System Software for Deep Learning, Fully Generated by AI Agents

Vulgarisation

Introduction

Deux mois. C'est le temps qu'il a fallu à une équipe de NVIDIA pour produire un logiciel complet de deep learning — moteur de tenseurs, gestion mémoire GPU, différentiation automatique, interfaces Python et JavaScript — sans qu'un humain ne relise manuellement une seule ligne de code générée. Le projet s'appelle VibeTensor, et il pose une question frontale à toute l'industrie du logiciel : jusqu'où peut-on déléguer la construction de systèmes complexes à des agents IA ?

VibeTensor n'est pas un jouet. Le dépôt pèse plus de 200 000 lignes de code, couvre du C++20 au CUDA en passant par Python et TypeScript, et tourne sur les GPU les plus récents de NVIDIA (H100 Hopper et Blackwell). Pourtant, chaque modification a été proposée par un agent de codage piloté par un grand modèle de langage, puis validée automatiquement par compilation et tests. Les chercheurs ont guidé le processus à haut niveau — « implémente un allocateur mémoire avec cache », « ajoute le support des CUDA Graphs » — mais n'ont jamais révisé les diffs un par un.

Le problème

Construire un framework de deep learning est un exercice d'ingénierie logicielle parmi les plus exigeants qui existent. PyTorch, le standard actuel, mobilise des centaines de développeurs depuis près d'une décennie. Le code traverse des couches d'abstraction radicalement différentes : une API Python élégante pour le chercheur, un runtime C++ performant pour le dispatch d'opérateurs, un moteur de différentiation automatique (autograd) qui trace les dépendances entre calculs, et enfin une couche CUDA qui gère les flux d'exécution, la mémoire et les graphes de calcul directement sur le GPU.

Chacune de ces couches est un projet logiciel en soi. Leur assemblage impose des contraintes croisées : l'allocateur mémoire doit coopérer avec le moteur autograd, qui doit synchroniser les flux CUDA, qui doivent respecter les invariants du dispatcher. Quand un bug apparaît, il peut traverser trois ou quatre couches avant de se manifester — un cauchemar pour le débogage humain, et a fortiori pour une IA.

La question posée par l'équipe NVIDIA est donc précise : des agents de codage peuvent-ils produire un système logiciel qui traverse toutes ces frontières d'abstraction, du Python jusqu'à la gestion mémoire GPU, en restant cohérent ?

La méthode

L'équipe a traité les agents de codage comme des boîtes noires. Pas de framework d'agent maison, pas de technique secrète. Des LLM capables de générer du code, équipés d'outils pour compiler, exécuter des tests et comparer les résultats avec une implémentation de référence (PyTorch, en l'occurrence). Les humains fournissaient des objectifs — « implémente la rétropropagation pour les tenseurs CUDA » — et fixaient des invariants à respecter, puis laissaient les agents itérer.

Le cycle de travail ressemblait à une boucle de développement classique, mais accélérée : spécifier un objectif, générer du code, compiler, tester, élargir la validation. Deux garde-fous se sont révélés indispensables. Le premier : des tests servant de spécifications exécutables. Pas seulement des tests unitaires isolés, mais des tests de régression multi-étapes qui simulent de vrais entraînements. Le second : des vérifications différentielles, c'est-à-dire la comparaison systématique des sorties de VibeTensor avec celles de PyTorch, opérateur par opérateur, pour détecter toute divergence numérique.

L'équipe a aussi utilisé de la revue de code multi-agents : plusieurs agents examinaient le travail des autres pour repérer les abstractions redondantes, les cas manquants ou les patterns dangereux. Une sorte de pair programming entre IA.

L'analogie la plus parlante : imaginez qu'on demande à un architecte de concevoir un immeuble pièce par pièce, sans jamais voir le plan d'ensemble, mais en vérifiant après chaque étape que les murs tiennent et que la plomberie fonctionne. Chaque pièce passe l'inspection locale, mais personne ne garantit que l'ensemble forme un bâtiment harmonieux. C'est exactement le défi auquel les agents ont été confrontés — et la source des problèmes les plus intéressants du projet.

Architecture du système généré

VibeTensor reproduit l'architecture en couches d'un framework de deep learning moderne. Au sommet, une interface Python qui imite l'API de PyTorch (création de tenseurs, opérations, entraînement) et une interface expérimentale en Node.js/TypeScript. Ces frontends communiquent avec un cœur C++20 via nanobind, une bibliothèque de bindings légère.

Le cœur comprend un système de tenseurs avec vues et strides (les tenseurs partagent la mémoire physique via un compteur de références), un dispatcher qui route les appels d'opérateurs vers les bons kernels selon le device (CPU ou CUDA), et un moteur autograd en mode inverse qui construit dynamiquement le graphe de calcul pendant l'exécution. La couche CUDA gère les streams, les events, un allocateur avec cache ordonné par stream, et le support des CUDA Graphs pour capturer et rejouer des séquences d'opérations GPU.

Le système propose aussi une ABI C stable pour charger dynamiquement des plugins d'opérateurs, l'interopérabilité via DLPack (échange de tenseurs sans copie mémoire entre frameworks), et des hooks pour intégrer des kernels écrits en Triton ou avec CUTLASS.

Résultats clés

Le dépôt final contient 940 fichiers source et plus de 201 000 lignes de code (hors dépendances tierces). La suite de tests regroupe 419 fichiers — 194 en C++ et 225 en Python — pour environ 54 000 lignes de tests. Le ratio tests/code est élevé, ce qui reflète la stratégie « les tests comme spécifications ».

Sur le plan des performances brutes des kernels individuels, les résultats sont encourageants. Un kernel de RMSNorm généré par IA atteint un speedup de 6.3× par rapport à l'implémentation PyTorch sur H100. Le kernel de Rotary Embedding affiche 5.3×. Pour l'attention fusionnée en configuration NanoChat (batch 32, séquence de 2048 tokens), le kernel Triton dépasse FlashAttention de 54% en forward et 26% en backward.

Mais les performances bout-en-bout racontent une histoire différente. Sur trois tâches d'entraînement (inversion de séquence, ViT sur CIFAR-10, miniGPT sur Shakespeare), VibeTensor est 1.7 à 6.2 fois plus lent que PyTorch. Le système converge correctement — les courbes de loss et d'accuracy suivent la même trajectoire que PyTorch — mais chaque itération prend beaucoup plus de temps.

L'écart varie selon l'architecture GPU. Sur Blackwell, la tâche d'inversion de séquence affiche un ralentissement de 1.7× seulement, contre 3× sur Hopper. Le miniGPT passe de 5.8× (Hopper) à 4× (Blackwell). La tendance est cohérente : le matériel plus récent réduit l'écart, mais ne le comble pas.

L'équipe a aussi testé le passage à l'échelle multi-GPU sur Blackwell, avec un plugin de ring-allreduce basé sur CUTLASS. Le débit passe de 2.19 millions d'échantillons par seconde sur un GPU à 3.69 millions sur quatre GPU, un facteur d'environ 1.7×. Loin du scaling linéaire, mais le sous-système expérimental « Fabric » n'est pas optimisé pour ça : il démontre que les chemins de synchronisation inter-GPU fonctionnent de bout en bout.

L'effet Frankenstein

Le résultat le plus instructif du projet n'est pas un chiffre de performance. C'est l'identification d'un mode de défaillance propre aux systèmes générés par IA, que les auteurs baptisent « effet Frankenstein ». Chaque sous-système, pris isolément, est correct. L'allocateur gère bien la mémoire. L'autograd calcule les bons gradients. Le dispatcher route les opérateurs sans erreur. Mais leur assemblage produit un système globalement sous-optimal.

L'exemple concret : le moteur autograd utilise un verrou global (mutex) qui empêche deux passes backward de s'exécuter simultanément. Ce choix est raisonnable du point de vue de la correction — il évite les deadlocks et les conditions de course. Mais il sérialise le travail côté CPU, ce qui affame les kernels GPU : des kernels performants passent leur temps à attendre que le CPU leur soumette du travail. Le résultat : des GPU sous-utilisés malgré des kernels rapides.

Ce phénomène est structurel. Un agent qui génère l'autograd optimise pour la correction locale. Un autre agent qui génère les kernels optimise pour la performance locale. Personne — ni humain, ni agent — n'a encodé la contrainte de performance globale dès le départ. Les tests vérifient que chaque pièce fonctionne, pas que l'ensemble performe.

Implications pratiques

VibeTensor ne remplacera pas PyTorch. Les auteurs le disent eux-mêmes : le système est un prototype de recherche, avec une surface d'API incomplète, des performances non optimisées, et des risques de sécurité inhérents au code généré par machine. Ils déconseillent explicitement tout usage en production.

L'intérêt est ailleurs. Ce projet fournit un banc d'essai grandeur nature pour étudier comment les agents IA se comportent face à des systèmes logiciels multi-couches. Les leçons se transposent directement à d'autres domaines : systèmes d'exploitation, bases de données, compilateurs — tout logiciel où des sous-systèmes doivent coopérer à travers des frontières d'abstraction.

Pour les équipes d'ingénierie, l'enseignement principal concerne l'infrastructure de validation. Les tests unitaires ne suffisent pas quand le code est généré par IA. Il faut des tests de composition — des scénarios qui exercent l'interaction entre sous-systèmes sur de longues séquences d'exécution. Un buffer GPU non initialisé qui fonctionne pendant 10 itérations et plante à la 11ème ne sera pas détecté par un test unitaire classique.

La revue de code multi-agents est un autre pattern transférable. Quand un seul agent génère du code, il tend à accumuler des abstractions redondantes et des conventions incohérentes. Confronter plusieurs agents produit un effet de relecture croisée qui attrape une partie de ces dérives — pas toutes, mais suffisamment pour maintenir la cohérence du système.

Le fait que NVIDIA publie ce projet en open source sous licence Apache 2.0 est aussi un signal. Le code généré par IA, avec ses défauts documentés, devient un objet d'étude partagé. D'autres équipes peuvent reproduire les benchmarks, identifier de nouveaux modes de défaillance, et tester leurs propres stratégies de validation.

Pour aller plus loin

L'effet Frankenstein soulève une question ouverte : comment encoder des contraintes de performance globale dans un workflow de génération agent par agent ? Les tests vérifient la correction. Les benchmarks mesurent la performance. Mais aucun garde-fou actuel ne détecte qu'un mutex global dans l'autograd va sérialiser le pipeline GPU trois couches plus bas. Des travaux sur la spécification formelle de propriétés non-fonctionnelles (latence, débit, utilisation GPU) pourraient combler ce vide.

La question de la maintenance est aussi sans réponse. Générer 200 000 lignes de code en deux mois, c'est spectaculaire. Mais faire évoluer ce code — corriger un bug qui traverse quatre couches, refactorer le dispatcher sans casser l'autograd — reste un défi non adressé. Le code généré par IA accumule ce que les auteurs appellent des « conventions incohérentes et des abstractions redondantes ». La dette technique s'installe dès la première génération.

Le gap de performance (1.7× à 6.2× par rapport à PyTorch) invite à explorer des stratégies de profiling automatisé intégrées au workflow de génération. Si l'agent pouvait identifier les goulots d'étranglement après chaque modification et proposer des optimisations ciblées, le système résultant serait probablement plus homogène en performance.

Enfin, VibeTensor ne couvre qu'un sous-ensemble du spectre d'un framework de deep learning moderne. L'entraînement distribué, la quantification, l'export de modèles, le support de multiples backends matériels — autant de directions où la génération par agents n'a pas encore été testée à cette échelle. Le chemin entre « un prototype qui converge sur CIFAR-10 » et « un framework utilisable par des milliers de chercheurs » reste long, et c'est précisément ce chemin qui fera l'objet des prochaines études.
